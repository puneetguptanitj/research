Data Platform Deep-Dive · Table Format Layer (L2)

Apache Iceberg: Open Table Format

A vendor-neutral open specification that layers a tree of immutable metadata files over columnar data files on object storage, giving any engine that reads that metadata tree ACID transactions, serializable snapshot isolation, time travel, and schema and partition evolution — without touching the data files.

Prepared 2026-05-23 Reading time ~29 min Primary source Apache Iceberg: The Definitive Guide (Shiran, Hughes, Merced · O'Reilly 2024) Focus Architecture · Landscape · Auth · Connections · v3

TL;DR

Apache Iceberg is an open table format specification (not an engine, not a catalog) that sits at L2 in the data platform stack. Its core insight: the only thing any catalog needs to store is a path to a current metadata file; everything else — schema history, partition specs, file statistics, snapshot lineage — lives in an immutable tree of Avro and JSON files collocated with the data on object storage. An atomic swap of that pointer is all it takes for ACID semantics. Any engine that can walk the tree is a first-class citizen. In 2025, the v3 spec added deletion vectors, row lineage, semi-structured VARIANT, nanosecond timestamps, and geospatial types, materially closing the gap with Delta Lake on high-update workloads.

  1. Metadata tree = the innovation. The chain catalog pointer → metadata file → manifest list (snapshot) → manifest files → data files gives Iceberg all its guarantees. Committing a new snapshot is a single atomic CAS on the catalog pointer; readers at the old pointer are unaffected.
  2. Engine-agnostic by design. Iceberg is a spec, not a library you depend on. Trino, Spark, Flink, DuckDB, Dremio, Athena, and Snowflake all read the same on-disk format. No engine owns the table.
  3. Hidden partitioning + partition evolution. Partition transforms (hour, day, bucket, truncate) are stored in the metadata file, not the query. Changing a partition strategy never rewrites historical files; new snapshots use the new spec while old manifests retain the old one.
  4. Iceberg has no built-in auth. Auth lives at two boundaries: (a) the IRC catalog authenticates callers via OAuth2/OIDC and vends short-lived, path-scoped storage credentials; (b) storage enforces IAM policies on those credentials. Iceberg itself only defines table properties for server-side encryption type selection.
  5. COW vs MOR, and why v3 matters. Copy-on-write rewrites affected data files on update (good read performance, high write amplification). Merge-on-read appends positional or equality delete files (low write cost, read-time join penalty at scale). Iceberg v3 deletion vectors — per-file bitmaps stored as Puffin blobs — largely eliminate the MOR read penalty without rewriting data files.
  6. Small files and compaction are the operational reality. Streaming ingestion creates many small files. Each small file adds a manifest entry; manifest scanning cost grows linearly. rewriteDataFiles compaction is mandatory for production tables with frequent writes, and it must run regularly or query performance degrades silently.

1What it is & core architecture

Apache Iceberg is a table format specification, first proposed at Netflix in 2017 and donated to the Apache Software Foundation. The spec defines exactly how bytes are organised on a filesystem or object store so that any conforming engine can read and write the same table. The spec has had three major versions: v1 (2018 — basic format), v2 (2020 — delete files, sequence numbers, row-level deletes), and v3 (2025 — deletion vectors, row lineage, geospatial, VARIANT, nanosecond timestamps).

An Iceberg table is organised into three logical layers. From bottom to top: the data layer holds the actual bytes; the metadata layer indexes those bytes with statistics and tracks changes as snapshots; the catalog layer holds a single pointer to the current metadata file. Every read starts at the catalog and walks down; every write builds new metadata and ends with an atomic catalog pointer update.

Data Layer

The data layer contains data files and delete files. Data files store actual rows in a columnar format — Parquet (by far the most common), ORC, or Avro. Iceberg is file-format agnostic; the format in use is recorded per-file in the manifest, not globally. Parquet is dominant because its row-group statistics (per-column min/max, null counts) feed directly into Iceberg's pruning chain: the engine reads column statistics from the manifest file first, opens the Parquet footer only if the manifest passes, and reads individual row groups only if the footer passes. Three independent filter stages on the way to reading a byte.

Delete files record which rows have been logically removed without rewriting the data file they belong to — this is the merge-on-read (MOR) strategy. The Iceberg v2 spec defines two kinds. Positional delete files identify a deleted row by (file_path, row_position) — a fast lookup but tightly coupled to the physical file. Equality delete files identify rows by the values of one or more columns (e.g., order_id = 1234) — more flexible, but at read time the engine must join every data file with every relevant equality delete file, causing scan amplification that grows with write frequency. Sequence numbers (a monotonically increasing counter per snapshot) prevent equality delete files from accidentally deleting a row added after the delete was written: an equality delete with sequence number N only applies to data files with sequence number < N.

The Iceberg v3 spec replaces both v2 delete-file types with deletion vectors: a compact bitmap (one bit per row in a data file) stored as a Puffin blob attached to the data file. Deletion vectors eliminate the equality-delete join entirely — a reader ANDs the bitmap with the row data in a single pass. AWS announced support for Iceberg v3 deletion vectors in November 2025; Databricks Runtime 18.0+ and Snowflake's open catalog are also in public preview as of early 2026.

Metadata Layer

The metadata layer has three file types, all stored alongside data files in object storage.

Manifest files (Avro format) are the leaves of the metadata tree. Each manifest file tracks a subset of data or delete files (never both in the same manifest). For every tracked file it records: the file path, format, partition membership, record count, and per-column statistics (lower bound, upper bound, null count, NaN count). These statistics are written by the engine that writes the data — cheaply, because the writer has already processed the rows. This is in contrast to the Hive table format, where statistics are gathered by expensive post-write scan jobs that are rarely re-run. In practice, Iceberg tables carry accurate, up-to-date statistics; Hive tables often do not.

Manifest lists (Avro format) represent a snapshot of the table at a point in time. Each manifest list entry references one manifest file and includes the manifest's partition column ranges and row count deltas — enough to skip the entire manifest without opening it if a query's partition predicates don't overlap. The manifest list is what Iceberg calls a snapshot. Its sequence_number field is the snapshot's monotonic commit order.

Metadata files (JSON format) are the root of the tree. A metadata file records the table UUID, the full schema history (as a list of schema objects with stable schema-ids), all partition specs ever used (each with a stable spec-id), all known sort orders, a list of valid snapshots, and the current-snapshot-id. Every commit to the table writes a new metadata file; the old one is never mutated. This immutability is what enables time travel: any old metadata file is still valid, and reading it gives a consistent view of the table as it was at that snapshot.

Puffin files store statistics and index structures that exceed what Parquet footers or manifest statistics can express. A Puffin file contains arbitrary binary blobs tagged with a type string. Currently two blob types are in active use: apache-datasketches-theta-v1 (a Theta sketch for approximate distinct-value counts, enabling fast COUNT(DISTINCT ...)), and — new in v3 — deletion vector blobs (deletion-vector-v1) stored as compressed bitmaps. Puffin files are referenced from the metadata file's statistics array and are associated with a specific snapshot.

Key mechanism

A commit is valid if and only if no other writer updated the catalog pointer between the time this writer read it and the time it tries to swap it. Every catalog implementation must provide compare-and-swap (CAS) semantics on the pointer. If the CAS fails, the writer discards its new metadata files and retries from scratch — this is optimistic concurrency control (OCC), not locking.

Catalog Layer

The catalog holds exactly one piece of information per table: the path to the current metadata file. Everything else Iceberg needs is in that file or its descendants. The catalog does not need to understand Iceberg's metadata structure — it just needs to be able to atomically update a string value. This is why Hive Metastore, AWS Glue, DynamoDB, a filesystem version-hint.txt, or a full REST catalog (IRC) can all serve as an Iceberg catalog: they only differ in how they guarantee atomicity of the pointer swap. The REST Catalog (IRC) moves this pointer swap to a server-side commit endpoint (POST /v1/{prefix}/namespaces/{ns}/tables/{table}), which centralises conflict detection and enables additional features like credential vending.

2Placement in the data platform & AI landscape

Iceberg occupies L2 (Table Format) in the platform stack. It is logically above L0 (object storage, which hosts its files) and L1 (file formats, which it manages), and below L3 (catalogs, which track its metadata pointer) and L4 (query engines, which walk its tree). It is the only layer whose primary job is to impose ACID semantics and a table abstraction over a collection of raw files.

The key distinction: Iceberg is a spec for how files are organised, not a service you run. You cannot "connect to Iceberg" — you connect to a catalog that implements the IRC spec, which then points you at Iceberg metadata files on object storage. This means Iceberg is simultaneously everywhere (any engine, any cloud) and nowhere (no process, no port). Its only runtime presence is files on disk.

What it replaces: The Hive table format, where a single Hive Metastore tracks partition directories rather than individual files, has no snapshot isolation, requires table locks for concurrent writes, and cannot evolve schemas or partitions without data rewrites. Iceberg removes all of those constraints.

What it competes with: Delta Lake (created at Databricks, OSS under Linux Foundation) and Apache Hudi (created at Uber, optimised for write-heavy streaming). All three solve the same core problem. Delta Lake uses a JSON transaction log (_delta_log/) rather than Iceberg's Avro manifest tree — slightly different consistency properties at the transaction log level. Hudi adds a custom index for upsert performance. As of 2025–2026, Iceberg has the broadest multi-engine adoption and the strongest vendor-neutral governance (ASF), making it the default choice for new open lakehouse deployments not tied to a single vendor's ecosystem.

3How it works

Write lifecycle

A write to an Iceberg table follows six steps. The engine first reads the current state from the catalog (the snapshot it is building on top of). It then writes data files to object storage — in parallel, with no coordination. Once data files are written, it writes a manifest file tracking those files and their statistics. It reads the current manifest list, constructs a new manifest list that includes the old manifests plus the new one, and writes that as a new snapshot. It writes a new metadata file with that snapshot added and current-snapshot-id updated. Finally it issues a CAS commit to the catalog: "if the current pointer is still X, set it to Y." If another writer committed in the meantime, the CAS fails; the engine discards its new metadata files and retries from step one, re-reading the updated state.

This is optimistic concurrency control with no global lock. Two writers can proceed concurrently; the second to commit wins; the first retry fast. In practice, this works well for batch and moderate-frequency streaming workloads. At very high commit rates (hundreds per second to the same table), OCC retry overhead becomes significant — a regime for which Iceberg is not designed.

Read lifecycle and the pruning chain

A read query passes through three independent pruning stages before any data bytes are transferred. The engine reads the catalog to get the current metadata file path. It reads the metadata file to identify the current snapshot's manifest list. It scans the manifest list and discards any manifest file whose partition column ranges (stored in the manifest list entry) do not overlap the query's partition predicate — this is partition pruning, and it happens without opening any manifest file. For the surviving manifest files, it applies column min/max statistics to discard data files — this is data-file pruning. For surviving data files (Parquet), it applies Parquet row-group statistics for a third round of pruning before reading any row. Only then is actual data transferred.

For MOR v2 tables, after identifying surviving data files, the engine also identifies relevant delete files. Positional delete files tell it which row indices to skip. Equality delete files require a join: for each data file, the engine must check whether any row matches any active equality delete. This join is the MOR v2 read penalty. For MOR v3 (deletion vector) tables, the engine simply fetches the bitmap blob for each data file and filters the row scan — no join, no cross-file lookup.

Schema and partition evolution

Schema evolution in Iceberg is column-ID based, not name-based. Each column carries a stable integer field-id that persists across renames, reorders, and drops. An engine reading a file uses the field-id from the file's schema, not from the current table schema, to map columns — so adding a column never reads garbage values from old files, dropping a column leaves old file bytes untouched, and renaming is safe because the ID is the canonical identity.

Partition evolution works via partition spec versioning. The metadata file stores a list of all partition specs ever used, each with a unique spec-id. Each manifest file records which spec was in effect when its data files were written. When the engine plans a query, it evaluates the partition predicate against each spec independently: newer data files use the new spec; older data files use the old spec. Neither set needs to be rewritten. This is hidden partitioning: the query does not need to know that HOUR(ts) replaced DAY(ts); Iceberg handles the translation transparently.

Key insight

Partition evolution is why Iceberg tables can be repartitioned at zero cost. A table that started with PARTITION BY day(event_time) can gain a PARTITION BY hour(event_time) spec for future data without touching the existing 3TB of historical files.

Branching and tagging

Iceberg v2 introduced named branches and tags as references in the metadata file's refs map. A branch is a mutable named pointer to a snapshot (updated on each new commit to the branch); a tag is an immutable named pointer. Branches enable write isolation — two concurrent writers can commit to separate branches and merge by reference swap — and are the mechanism underlying Nessie's multi-table git-style versioning. The main branch is always called main and is equivalent to the current-snapshot-id.

4Authentication & identity

Iceberg the spec has no authentication or authorization built in. It defines table properties for server-side encryption type selection (write.object-storage.enabled, s3.sse.type), but these are hints to the writer, not enforcement. Security in an Iceberg lakehouse is provided at three independent boundaries, applied in sequence.

Boundary 1 — Object storage IAM

Every read and write of a data or metadata file goes through the object storage layer (S3, GCS, ADLS). S3 enforces IAM policies, bucket policies, and ACLs; GCS enforces IAM with service account bindings; ADLS Gen2 uses Azure RBAC plus ACLs at the filesystem level. Encryption at rest is handled here: S3 SSE-S3 (AES-256, S3-managed keys), SSE-KMS (AWS KMS-managed keys, required for compliance in most regulated environments), or SSE-C (customer-provided keys, maximum control). None of these are Iceberg-specific — they apply to every byte on the storage path regardless of table format.

Boundary 2 — IRC Catalog (Policy Decision Point)

The Iceberg REST Catalog (IRC) is where table-level authorization happens and where the shift from long-lived to short-lived credentials occurs. The engine authenticates to the IRC using an OAuth2 Bearer token. The IRC validates the token (signature, expiry, audience), evaluates whether the authenticated principal has the requested privilege (typically RBAC — namespace-level or table-level grants, with some IRC implementations adding row/column-level ABAC), and if authorized, calls the underlying cloud IAM to generate vended credentials: short-lived, path-scoped object-storage credentials (AssumeRoleWithWebIdentity on AWS returns a temporary STS access_key_id / secret_access_key / session_token scoped to the table's S3 prefix; GCS returns a service account access token). The IRC returns these vended credentials alongside the table metadata in the LoadTableResponse.config field. The engine then reads S3 directly using the vended credentials — it never holds a long-lived storage key, and the credentials expire (typically in 1 hour or less). Engines signal they support credential vending by sending the header X-Iceberg-Access-Delegation: vended-credentials in the loadTable request.

Breaking change — Iceberg 1.6.0

The IRC's own OAuth token endpoint (POST /v1/oauth/tokens) was deprecated in Iceberg (Java) 1.6.0 and is planned for removal in Iceberg 2.0. Clients must configure oauth2-server-uri to point to an external OAuth2/OIDC identity provider rather than relying on the catalog to issue tokens. Deployments that use the built-in endpoint will need to migrate to an external IdP (Keycloak, Okta, Azure AD, AWS Cognito) before the Iceberg 2.0 upgrade path.

Boundary 3 — Query engine

Users authenticate to the query engine separately. Trino supports OAuth2/OIDC, JWT, LDAP, Kerberos (GSSAPI), and client certificate/mTLS — all requiring TLS on the coordinator HTTPS port. Spark uses Kerberos for cluster-level auth or delegates to the notebook/job environment. The engine then impersonates (or proxies as) the user when calling the IRC — either using the user's own OAuth token directly, or using a service account token with the user identity passed as a claim for ABAC evaluation at the catalog.

Note

Not all catalog implementations support credential vending — it requires the catalog to have an IAM role it can assume on behalf of the caller. The Hadoop catalog (filesystem-based) has no auth at all and is not production-safe. AWS Glue's IRC endpoint and Apache Polaris support vended credentials; verify capability at the specific catalog implementation you deploy.

6Trade-offs, failure modes & when (not) to use it

Strengths

Genuine multi-engine neutrality. The Iceberg spec is owned by the Apache Software Foundation. No single vendor controls the format. Trino, Spark, Flink, DuckDB, Dremio, Athena, BigQuery Omni, and Snowflake External Tables can all read the same on-disk format. In practice, this matters most when you need to mix compute engines (e.g., Spark for heavy ETL, Trino for interactive BI, Flink for streaming ingestion) over a single data copy.

Schema and partition evolution at zero data cost. Column-ID–based schema evolution and partition-spec versioning mean no rewrites for schema or partitioning changes — a significant operational advantage over Hive format and over Delta Lake, which uses LIQUID CLUSTERING to avoid this but requires a managed service in practice.

ACID without a coordinator process. OCC via CAS on the catalog pointer gives serializable snapshot isolation with no long-running lock manager. Readers are always non-blocking. Concurrent writers are isolated until commit.

Time travel and audit. Because every snapshot is immutable and the metadata file tracks the full snapshot log, any historical state is reachable by snapshot ID or by timestamp (within the snapshot expiry window). This is free — it requires no ETL, no CDC pipeline, no separate audit table.

Failure modes and sharp edges

Small files — the primary operational hazard. Streaming ingestion produces many small data files per micro-batch. Each small file adds a manifest entry; reading a table with 10 million small files requires scanning manifests for 10 million entries before pruning, even if the query touches 100. The solution is regular compaction (rewriteDataFiles in the Iceberg Java API), but compaction has write amplification — it reads and rewrites existing data files — and must be tuned to run frequently enough to prevent manifest bloat without competing with live ingestion. Failing to run compaction is the single most common reason a production Iceberg table degrades over weeks.

Equality delete scan amplification (v2 MOR). An equality delete file applies to all data files in the same partition with a lower sequence number. If a table has 1,000 data files in a partition and accumulates 50 equality delete files, reading any row from that partition requires checking 50 equality delete files against each of 1,000 data files — a cross product of 50,000 file interactions per partition, per query. This grows quadratically with both data file count and uncompacted delete file count. v3 deletion vectors eliminate this; for v2 MOR tables, aggressive compaction (which rewrites the data files, absorbing the deletes) is the only mitigation.

Catalog pointer CAS contention. OCC assumes low conflict. In workloads where hundreds of concurrent writers all target the same table (e.g., a high-throughput streaming pipeline with many Flink tasks committing independently), CAS retries multiply. Each retry discards new metadata files and rebuilds them — CPU and I/O cost without data movement. The mitigation is to batch writes into fewer, larger commits (e.g., Flink's checkpoint.interval tuning) or use a branching model (commit to sub-branches and merge).

Metadata file accumulation. Every commit writes a new metadata file. A table receiving 100 commits/day accumulates 36,500 metadata files per year. The write.metadata.delete-after-commit.enabled table property enables automatic cleanup, keeping only the last N metadata files. Without this, the metadata directory grows unboundedly, and the metadata log entries in each file also grow, increasing the parse cost of each catalog read.

No native multi-table transactions in the spec. Iceberg v2 commits are atomic per table. Cross-table atomicity (e.g., "deduct from table A and credit to table B in a single atomic transaction") requires a catalog that implements multi-table branching (Nessie's git-model, or Unity Catalog's multi-table commit API). This is a real gap for use cases that need referential integrity across Iceberg tables.

When to use Iceberg

Iceberg is the right choice when you need open, multi-engine access to a governed analytical table at scale. The canonical pattern: Flink or Spark for streaming/batch ingestion, Trino for interactive BI, and a governance catalog for lineage and classification — all over one copy of data, with ACID guarantees and no vendor lock-in on the table format. It is also appropriate when schema or partition evolution will be frequent, since the zero-rewrite guarantees are a genuine operational advantage. Time travel and audit requirements (regulatory, debugging) are well-served by the immutable snapshot model.

When not to use Iceberg

Iceberg is a poor fit for high-frequency transactional workloads. If you need sub-second commit latency at hundreds of writes per second to a single table, the OCC model and metadata overhead will not keep up — use an operational database (Postgres, Lakebase) instead. For very small data (millions of rows, gigabytes), the metadata overhead per table (catalog, manifest files, metadata files) adds operational complexity without proportional benefit. For environments where running a catalog is not feasible, the Hadoop filesystem catalog is operationally unsafe on S3 (no atomic rename → data loss risk on concurrent writes); you must run a real catalog.

Production checklist

Run compaction (rewriteDataFiles) on a schedule keyed to your write frequency. Enable write.metadata.delete-after-commit.enabled=true and write.metadata.previous-versions-max=100. Run snapshot expiry (expireSnapshots) to bound time-travel retention and allow orphan file GC. Monitor manifest count per partition — above ~10,000 manifest entries per partition, query planning latency will be noticeable.

7Architect's take

Iceberg has won the table format war for open platforms. That's not marketing — it's the observable state of ecosystem alignment as of 2026. AWS S3 Tables, Snowflake Open Catalog, Azure's preview support, Google Cloud's Lakehouse Iceberg REST Catalog, and every major query engine shipping Iceberg connectors is not a coincidence. The ASF governance and the clean separation of the table spec from the catalog spec (IRC) are the structural reasons. The practical implication: new analytical table investments on open storage should default to Iceberg unless there is a specific, justified reason to prefer Delta Lake (e.g., a deeply Databricks-native stack where Liquid Clustering's auto-optimize is operationally valuable).

The IRC catalog is the real control plane decision. Where you run your catalog (Apache Polaris, Unity Catalog OSS, Apache Gravitino, Lakekeeper, Nessie, AWS Glue) determines your auth model, your credential vending capability, your multi-table transaction support, and your governance integration surface. Iceberg's own on-disk format is stable; the catalog ecosystem is still consolidating. In 2026, Apache Polaris is the ASF-native choice; Databricks Unity Catalog is the most feature-complete for Databricks shops; Nessie is the right answer for teams that want git-style branching semantics. Pick based on operational maturity and governance requirements, not marketing.

v3 deletion vectors close the Delta Lake MOR gap. If your workload is high-update streaming (CDC pipelines, event corrections, late-arriving records), v2 Iceberg MOR was a pragmatic trade-off against a growing read penalty. v3 deletion vectors change that equation: the per-file bitmap approach has read overhead proportional to the bitmap size, not to the number of deletes — it scales cleanly. As of mid-2026, v3 is in public preview on AWS (S3 Tables), Databricks, and Snowflake; Trino and Flink v3 support is in active development. Evaluate whether your catalog and engine support v3 before planning for it.

Watch over the next 12–18 months: (1) Iceberg v3 full ecosystem adoption — particularly Trino and Flink native deletion vector support; (2) IRC catalog consolidation — Polaris vs. Gravitino vs. Lakekeeper for the ASF-native slot; (3) multi-table transaction support becoming a standard IRC feature, which would remove Iceberg's last major limitation relative to warehouse systems; (4) Iceberg as the substrate for operational tables (Lakebase / OLTP-on-Iceberg) — the boundary between the analytical and operational lakehouse is actively dissolving.

Glossary

CAS (Compare-and-Swap)
An atomic operation: "if the current value equals X, set it to Y; otherwise, report failure." The catalog's CAS on the metadata file path is what gives Iceberg ACID.
Credential vending
The IRC catalog's mechanism for returning short-lived, path-scoped cloud storage credentials (AWS STS temporary keys, GCS access tokens) alongside table metadata, so engines never hold long-lived storage keys.
Deletion vector (v3)
A per-data-file bitmap stored as a Puffin blob. Each bit corresponds to a row; set bits mark deleted rows. Replaces positional and equality delete files in Iceberg v3.
Equality delete file (v2)
A file specifying rows to logically delete by matching column values (e.g., order_id = 1234). Applied at read time by joining against every data file with a lower sequence number in the same partition.
Hidden partitioning
Iceberg stores partition transforms (e.g., HOUR(ts)) in the metadata file; queries filter on the base column (WHERE ts > …) and Iceberg prunes partitions transparently, without the query knowing the physical layout.
Manifest file
An Avro file tracking a subset of data or delete files for one snapshot, including per-file column statistics. Written incrementally by each writer.
Manifest list
An Avro file listing all manifest files for one snapshot, with per-manifest partition column ranges for partition-level pruning. Equivalent to "a snapshot."
Metadata file
A JSON file at the root of the metadata tree. Contains table UUID, schema history, partition spec history, snapshot list, and pointer to current snapshot. Written on every commit; never mutated.
OCC (Optimistic Concurrency Control)
A concurrency strategy where writers proceed without locking, checking for conflicts only at commit time via CAS. Write conflicts result in retry, not blocking.
Partition evolution
Changing the partition strategy (adding, dropping, or transforming partition fields) without rewriting historical data files. New files use the new spec; old manifests retain the old spec-id.
Puffin file
A binary file format for statistical blobs associated with an Iceberg snapshot. Used for Theta sketches (approximate NDV) and deletion vector bitmaps (v3).
Sequence number
A monotonically increasing integer assigned per snapshot. Delete files apply only to data files with strictly lower sequence numbers, preventing tombstone resurrection.
