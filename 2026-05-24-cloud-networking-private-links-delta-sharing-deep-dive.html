Data Platform Deep-Dive

Cloud Networking for Data Platforms

The full taxonomy of cloud network isolation and connectivity primitives — VPC peering, private links, transit gateways, NAT gateways, and how Databricks Delta Sharing collapses the cross-org networking problem to plain HTTPS.

Prepared 2026-05-24 Reading time ~30 min Focus Architecture · Mechanisms · Punch-hole patterns · Delta Sharing Clouds AWS · Azure · GCP (AWS primary)

TL;DR

Cloud VPC networking gives you a private, isolated network slice. The primitives you use to connect that slice to the world — or to other slices — each trade off scope, security, cost, and operational burden in distinct ways. Choosing the wrong one is how platforms end up with either an over-exposed attack surface or an $80k/year transit gateway bill for shuffling internal data.

Delta Sharing is architecturally interesting precisely because it sidesteps the cross-account/cross-cloud networking problem entirely: instead of requiring a PrivateLink or VPN between two organizations, it vends short-lived, scope-limited object-storage credentials over HTTPS. The recipient only needs to reach the Databricks control plane and cloud object storage — both of which are already reachable over the internet with TLS.

  1. Internet Gateway ≠ NAT Gateway ≠ VPN Gateway ≠ Transit Gateway. Each serves a different traffic direction and scope. Conflating them is the root of most network architecture mistakes.
  2. PrivateLink exposes a specific service, not a network. This is the critical distinction from VPC Peering: no transitive routing, no broad blast radius, works across overlapping CIDRs. Use it for SaaS patterns and data platform services (Databricks, S3 Interface Endpoint, Redshift).
  3. VPC Peering is fine for two VPCs; Transit Gateway is the answer beyond that. The N*(N-1)/2 peering math, combined with the 125-connection limit and no transitive routing, makes peering unworkable at mesh scale.
  4. "Punching a hole" has a right and a wrong way. The right way is PrivateLink or a tightly-scoped security group rule with your own service behind an NLB. The wrong way is opening a CIDR range in a NACL without a compensating security group, or enabling full VPC peering when you only need service-level access.
  5. Delta Sharing replaces PrivateLink as the connectivity model for cross-org data sharing. The protocol vends pre-signed URLs / STS tokens scoped to exactly the files the recipient queried — no network plumbing between orgs required. For fully private storage (no public endpoints), a Delta Sharing Network Gateway or NCC + PrivateLink pattern closes the gap.
  6. Gateway VPC Endpoints for S3 are free and should always be deployed in any private subnet that reads from S3. Using NAT Gateway for S3 traffic instead costs roughly $0.045/GB that could be zero.

1What it is & core architecture

A Virtual Private Cloud (VPC) is a logically isolated segment of a cloud provider's network, defined by one or more CIDR blocks (e.g., 10.0.0.0/16) that you own. Inside it, you create subnets — slices of that CIDR bound to a specific availability zone. Subnets are either public (route table has a default route to an Internet Gateway) or private (no IGW route; egress via NAT or none). Route tables, security groups (stateful L4 firewall per ENI), and NACLs (stateless L3/L4 firewall per subnet) control traffic within and across the VPC boundary.

The primitives that connect VPCs to other networks, or expose services between them, fall into six families:

The six families in one paragraph each

Internet Gateway (IGW). Attached one-to-one to a VPC. Performs 1:1 NAT between the public IPv4 address assigned to an EC2 instance and its private IP. Scales horizontally and is never a bottleneck. Traffic in both directions traverses the public internet. Subnets with a 0.0.0.0/0 → igw-xxx route table entry are "public." Every public-facing load balancer or bastion lives here. Remove the IGW route from any subnet where instances should not be directly reachable from the internet.

NAT Gateway. Allows private subnets to initiate outbound connections to the internet (or to other VPCs) without being reachable inbound. Two modes: public NAT (sits in a public subnet, has an Elastic IP, SNATs to that EIP, traffic exits to the internet); private NAT (no EIP, translates source IPs for VPC-to-VPC routing when CIDRs would otherwise overlap). NAT Gateways are zonal; for HA you need one per AZ. Cost: $0.045/hr + $0.045/GB (us-east-1, 2025). The most common waste in AWS bills is routing S3-bound traffic through a NAT Gateway instead of a Gateway Endpoint.

VPN Gateway & Direct Connect Gateway. VPN Gateway terminates IPSec/IKEv2 Site-to-Site tunnels to on-premises hardware or software VPN peers; max ~1.25 Gbps per tunnel, multiple tunnels for HA. Direct Connect Gateway attaches to private virtual interfaces on a dedicated fiber circuit — deterministic latency, higher bandwidth (1–100 Gbps), but weeks to provision and expensive. Both are used for hybrid cloud connectivity. Neither helps with cross-account or cross-org connectivity within AWS.

Transit Gateway (TGW). A regional hub router. VPCs and VPN connections attach to it; you define route tables per attachment, enabling fine-grained routing policies. TGW supports transitive routing — VPC A can reach VPC C through TGW even if they're not directly peered — which is the core problem VPC Peering cannot solve. Cross-region TGW peering links two regional hubs, enabling a global private backbone. Cost: $0.05/attachment-hour + $0.02/GB (us-east-1). For 10+ VPCs or complex topologies this is almost always cheaper than the N*(N-1)/2 peering mesh it replaces.

VPC Peering. A direct, bilateral routing relationship between two VPCs — same account, cross-account, or cross-region. No intermediary; traffic stays on the AWS backbone. The constraint: no transitive routing. If A is peered with B and B is peered with C, A cannot reach C through B; you need an A-C peering. Limit: 125 active peering connections per VPC. Best for simple, stable, two-VPC topologies where you want full network access (all ports/protocols) between the two — e.g., a data plane VPC peered with a shared services VPC. Never build a full-mesh peering beyond ~5 VPCs.

VPC Endpoints — Gateway vs. Interface. Gateway Endpoints are free, route-table-based entries that steer S3 and DynamoDB traffic through the AWS backbone — no public internet, no NAT Gateway, no ENI. Interface Endpoints (powered by AWS PrivateLink) are ENIs injected into your subnets with a private IP; DNS overrides make the service hostname resolve to that private IP. AWS rewrites traffic destined for the service endpoint in the hypervisor layer — the traffic goes directly from the consumer's physical host to the provider's NLB backend, with no intermediate proxy. PrivateLink providers expose their service behind a Network Load Balancer and create a VPC Endpoint Service (com.amazonaws.vpce.region.vpce-svc-xxx); consumers create Interface Endpoints pointing at that service name.

Key distinction

VPC Peering and Transit Gateway expose entire networks to each other — any ENI in VPC A can potentially reach any ENI in VPC B (modulo security groups). PrivateLink exposes a single service. This is why PrivateLink is the correct pattern for SaaS exposure, multi-tenant data services, and cross-account access to specific resources: the blast radius is one service, not one network.

Azure equivalents

Azure maps closely: Service Endpoints route traffic to Azure PaaS services over the Azure backbone but the service retains its public IP — effectively a policy-based route with no network isolation. Private Link / Private Endpoint injects the service into your VNet as a private IP (like an AWS Interface Endpoint) and requires a Private DNS Zone to override the public hostname. Azure Virtual Network Peering is the VPC Peering analog; Azure Virtual WAN is the Transit Gateway analog. One Azure nuance: VNet Service Endpoints are quick to set up and good enough for most intra-Azure scenarios, but they don't work cross-tenant; Private Endpoints do.

GCP equivalents

GCP's Private Service Connect (PSC) maps to PrivateLink. PSC is implemented via Google's Andromeda SDN — the data plane is fully distributed with no centralized proxy, and traffic is NAT'd at the destination host. A PSC endpoint creates a forwarding rule in the consumer VPC; the producer publishes a service via a load balancer backend. GCP also has VPC Service Controls, which is architecturally different from PSC: it creates an API-layer security perimeter around GCP services to prevent data exfiltration — complementary to PSC, not a substitute. VPC Network Peering is the GCP equivalent of VPC Peering, with the same no-transitive-routing constraint.

2Placement in the data platform & AI landscape

Cloud networking is not itself one of the L0–L9 data platform layers — it is infrastructure that underlies all of them. But it is far from neutral: every data platform design decision (where the catalog runs, whether the query engine can reach object storage without traversing the internet, how cross-org sharing works) is constrained or enabled by the networking primitives chosen. Think of cloud networking as the "L-1" plane — it determines what is reachable by L0 and above.

The practical intersections that matter for data platform architects:

  • L0 ↔ networking: How query engines reach S3/ADLS/GCS. Gateway Endpoints (S3) remove NAT cost. Interface Endpoints bring S3 fully private. VPC firewall rules on buckets enforce that only traffic from known CIDR/VPC Endpoint is accepted.
  • L3/L4 ↔ networking: Databricks workspace, Unity Catalog control plane, Trino coordinators — all should be reachable only via PrivateLink from consumer VPCs, not over public internet.
  • Cross-org sharing ↔ networking: The traditional model requires cross-account PrivateLink or VPC Peering + storage policy. Delta Sharing replaces this with an application-layer protocol.
  • L8 ↔ networking: Security groups are the first enforcement point. IAM roles control what an EC2/Lambda/container is allowed to do; security groups control what it can reach. Both are required.

3How each primitive works — mechanisms

VPC Peering — bilateral route injection

When you create a VPC Peering connection, AWS injects routes into the route tables of both VPCs pointing at the peering connection ID. Traffic flows directly between the two VPCs on the AWS backbone without any intermediate hop — no router appliance, no encryption overhead. The route table entries must be added manually (or via Terraform); they are not automatic. Security groups in the destination VPC apply normally; NACLs on both sides apply. Crucially, route tables are VPC-scoped: if VPC-B is peered with both VPC-A and VPC-C, there is no route in VPC-B's route table that lets A-originated traffic exit through the VPC-C peering — AWS explicitly prohibits transitive routing in peering. IP addressing must not overlap.

Transit Gateway — attachment-based hub routing

A TGW has one or more route tables. Each VPC, VPN, and Direct Connect attachment is associated with a route table and can propagate its CIDR to it. By default, a VPC attached to a TGW can reach all other attached VPCs — transitive routing works because the TGW is the transit router. You can create isolated route tables (one per VPC) to enforce fine-grained segmentation: a "prod" table that only accepts routes from prod attachments, a "dev" table that only accepts dev. Inter-region peering links two TGWs, extending the hub-and-spoke globally. TGW also terminates Site-to-Site VPN and integrates with Direct Connect Gateway for on-prem.

AWS PrivateLink — the hypervisor-layer shortcut

On the provider side: a service runs behind a Network Load Balancer. You create a VPC Endpoint Service referencing the NLB; AWS assigns it a service name. You optionally require acceptance of new endpoint connections (default: required for security — do not disable this). On the consumer side: you create an Interface VPC Endpoint specifying the service name; AWS creates one ENI per specified subnet, each with a private IP from that subnet's CIDR. AWS simultaneously creates a private DNS entry (a zone override) so that the service's public hostname (e.g., *.execute-api.us-east-1.amazonaws.com) resolves to the ENI private IP inside the VPC. Traffic from the consumer's EC2/Lambda goes to the ENI private IP; AWS rewrites it in the hypervisor and routes it directly to the NLB target — the public internet is never involved.

Key mechanism

PrivateLink's data plane is fully distributed (implemented in the Nitro hypervisor). There is no centralized proxy. Measured latency overhead vs. direct VPC access is typically under 100µs. The NLB in the provider VPC is not a bottleneck at typical enterprise data platform scale — it handles millions of connections per second.

Gateway VPC Endpoints — route table magic, zero cost

For S3 and DynamoDB only. When you create a Gateway Endpoint, AWS injects a "pl-XXXXXXXX" (prefix list) entry into the route tables you specify, routing all traffic to those service IP ranges through the gateway endpoint — bypassing NAT Gateway and internet. No ENI, no DNS change, no security group. Access is controlled by an endpoint policy (IAM-style JSON) and by the bucket policy (aws:SourceVpce condition key). The prefix list is managed by AWS and updated automatically as S3 IP ranges change.

How the DNS override works for PrivateLink

Watch out

Split-horizon DNS is a common failure mode. If "Enable Private DNS Name" is not set on the Interface Endpoint, or if the VPC does not have DNS resolution enabled, the application will resolve the service hostname to its public IP and either fail (if IGW is absent) or succeed over the internet (defeating the purpose). Always verify with dig <endpoint-hostname> from within the VPC — it must return a 10.x.x.x address, not a public IP.

4Punching holes in VPCs — when and how

"Punching a hole" is informal language for creating inbound access to a resource that would otherwise be unreachable from outside its subnet or VPC. The phrase covers a spectrum from a single security group rule change to full VPC peering. The decisions cascade: who needs access → what scope of network do they need → what is the minimum surface area.

When you must (and must not) punch holes

Scenario Do you need to punch a hole? Recommended approach
Another VPC in same account needs to call a specific service you host Yes — scope to service PrivateLink: expose via NLB + VPC Endpoint Service
Another AWS account (trusted partner) needs broad access to your VPC Yes — scope to network VPC Peering or TGW attachment with route table restrictions
10+ VPCs need to route to a shared service VPC Yes — scope to service PrivateLink from service VPC (avoids TGW cost) OR TGW centralized endpoint pattern
On-premises network needs to reach cloud resources Yes — scope to required resources VPN Gateway (quick, encrypted) or Direct Connect (predictable latency); route only specific prefixes
Another organization needs to read your data assets No — use Delta Sharing Delta Sharing (REST + pre-signed URLs); no network plumbing required
Lambda/ECS in private subnet needs to call AWS API endpoints (not S3) No or minimal Interface VPC Endpoint for the specific service (e.g., SSM, STS, ECR); avoids NAT Gateway cost
Serverless workloads need access to private S3 No (for AWS-same-account) Gateway Endpoint (free); Interface S3 Endpoint if stricter isolation needed
External internet user needs to reach a private resource No — add indirection ALB in public subnet fronting private targets; do not open NACL/SG to 0.0.0.0/0 directly on private resources

The right way to expose a service across VPCs

The canonical pattern is: service behind a Network Load Balancer → VPC Endpoint Service → consumers create Interface Endpoints. The NLB is required because PrivateLink's data plane connects to NLB targets, not to EC2 instances directly. The security group on the NLB target EC2s/containers should allow inbound only from the NLB security group (or NLB's CIDR), not from 0.0.0.0/0. This way the only path to the service is through the NLB → PrivateLink → consumer ENI.

Security Group vs NACL — which to touch

Always prefer security group rules over NACL rules for punching access holes. Security groups are stateful (return traffic is automatically allowed), attached per-ENI (so rules are scoped to exactly the affected resource), and support source-as-security-group-ID referencing (which eliminates IP management entirely for same-VPC resources). NACLs are stateless — you must add rules for both inbound and outbound traffic, and they apply to all traffic entering/leaving the subnet, not just the resource you care about. Use NACLs as a defense-in-depth second layer (e.g., "block all traffic from this subnet to the database subnet except on port 5432"), not as the primary access control.

Common mistake

Opening a NACL rule for 0.0.0.0/0 on a port "temporarily" and forgetting it. NACLs have no stateful tracking — a rule that's open is open to all traffic matching that pattern. Security groups with specific source security group IDs are both safer and easier to audit.

Zero-trust alternatives to "punching holes"

For connectivity patterns that don't fit neatly into VPC primitives — especially ephemeral developer access, third-party SaaS-to-private-VPC connections, or multi-cloud without a VPN — consider identity-aware proxies and mesh tools: AWS Verified Access (OIDC/SAML before reaching a resource, no VPN needed), Tailscale (WireGuard-based overlay mesh, peer-to-peer with DERP relay, works across clouds), or Cloudflare Tunnel (outbound-only connector from the private resource, no inbound firewall rules needed). These shift the trust model from "network location" to "identity + device posture" — better for humans and for service-to-service connections that cross trust domain boundaries.

5Authentication & identity

Cloud networking itself is not an authentication system — it controls reachability, not identity. But every networking primitive interacts with identity systems in specific ways, and confusing network-layer access control with application-layer authentication is a common security flaw.

PrivateLink: network approval ≠ authentication

When a consumer creates an Interface Endpoint to your service, PrivateLink requires the provider to accept the connection (or configure auto-accept). This is a network-level admission control — it answers "is this VPC Endpoint ID allowed to reach my NLB?" It does not authenticate the application calling the service. Any EC2 in the consumer VPC that can reach the Interface Endpoint ENI will get a TCP connection to the provider's NLB. The service itself must enforce authentication — typically via TLS mutual authentication (mTLS) or Bearer tokens. Treat PrivateLink acceptance as a coarse network gate, not a substitute for app-level authn.

IAM for Gateway Endpoints

Gateway Endpoints for S3 can carry an endpoint policy — an IAM resource policy attached to the endpoint that restricts what operations and which S3 buckets the VPC can access through it. Combined with a bucket policy using aws:SourceVpce, you get: (a) only traffic through the endpoint reaches the bucket, and (b) only the operations in the endpoint policy are permitted. This is the correct pattern for locking down data lake S3 buckets to only internal VPC access.

Databricks PrivateLink: workspace-level network controls

Databricks exposes its workspace, the SCC relay, and the REST API all via PrivateLink endpoints. The workspace URL resolves to the Interface Endpoint ENI in your VPC. Authentication to Databricks itself is separate: OAuth2 + OIDC (Databricks identity, or federated via your IdP), enforced at the Databricks control plane after the network-level connection is established. Unity Catalog enforces RBAC/ABAC on top of that. In other words, PrivateLink handles network reachability; Unity Catalog + Databricks authn handles identity and authorization. Neither replaces the other.

Subtle risk

Enabling PrivateLink for a Databricks workspace but leaving the workspace's public access enabled ("no-public-access = false") is not full isolation. Set no-public-access = true to ensure the workspace API is only reachable through the PrivateLink endpoint, not over the internet. This is the Databricks workspace-level analog of setting a bucket policy with aws:SourceVpce.

6Delta Sharing vs. private links — the architectural substitution

The traditional approach to cross-account or cross-org data sharing in the cloud is: (1) set up VPC Peering or PrivateLink between the two accounts, (2) configure storage bucket policies to allow the recipient's IAM role, (3) share IAM credentials or assume-role trust. This creates a permanent network-level relationship between two organizations — expensive to set up, fragile to maintain, and operationally coupled (provider network changes break recipient access).

Delta Sharing decouples data sharing from network topology. It is an open HTTP REST protocol (published on GitHub; VLDB 2025 paper: Puttaswamy et al.) that works as follows: the data provider runs a Delta Sharing server (Databricks' is built into Unity Catalog). The server maintains a catalog of shares, recipients, and access grants. When a recipient queries a share, the server returns pre-signed URLs (AWS S3 pre-signed URLs, Azure SAS tokens, or GCS signed URLs) or STS tokens that grant temporary, scoped, read-only access to exactly the Parquet/Delta files needed for that query. The recipient reads directly from object storage using those credentials — no permanent network path to the provider's VPC is required.

How Delta Sharing vends credentials — the credential flow

When Delta Sharing does NOT eliminate network configuration

Delta Sharing simplifies networking for the common case (internet-accessible object storage). Three scenarios still require network configuration:

1. Provider storage has no public access. If the S3 bucket policy denies all public access and only allows the provider's VPC Endpoint, then pre-signed URLs generated by the Delta Sharing server will return HTTP 403 when the recipient tries to use them from outside the VPC. Solutions: (a) use a Delta Sharing Network Gateway (new in Summer 2025 — provider-side HTTPS proxy that re-serves the data, removing the need for recipients to reach the raw bucket), or (b) configure the bucket to allow traffic from the Delta Sharing server's IP ranges / VPC endpoint for the presign operation.

2. Fully private cross-cloud sharing (e.g., AWS recipient → Azure provider). If the provider's ADLS storage is private, the recipient's serverless compute needs a path. The pattern (Databricks, Jan 2025): configure an NCC (Network Connectivity Configuration) for serverless with a PrivateLink rule pointing at a VPC Endpoint Service in AWS, which routes via an NLB whose target group has the ADLS private endpoint IP. Cross-cloud: a Site-to-Site VPN or ExpressRoute/Direct Connect links AWS and Azure first. The NCC object tells Databricks serverless to use the PrivateLink path rather than the public internet for that hostname.

3. Recipient has strict egress controls. If the recipient's network blocks outbound internet but they need to reach the Databricks Delta Sharing endpoint, they need an Interface Endpoint for the Databricks workspace on the provider side, or they need their own Databricks workspace's NCC to route sharing traffic through a private path.

The key insight

Delta Sharing inverts the networking burden: instead of both organizations building a persistent bilateral network relationship (VPC Peering, PrivateLink, VPN), the protocol treats the internet as the transport and uses object storage's own pre-signed URL/STS mechanism as the access control. The networking problem collapses to "can the recipient make outbound HTTPS connections?" — which is almost always yes.

The N*(N-1)/2 peering problem, illustrated

With 10 VPCs in a full mesh, you need 45 peering connections, each requiring route table entries on both sides (90 entries), and security group rules referencing 9 peer CIDRs per VPC. If any two VPCs' CIDRs overlap (common in org mergers or multi-team accounts), the entire mesh breaks. Transit Gateway cuts this to 10 attachments, one route table, and works regardless of CIDR overlap with careful route table design.

Delta Sharing egress costs

Delta Sharing within a region incurs no egress cost from the cloud provider's perspective — the data files are accessed from the same region. Cross-region or cross-cloud sharing incurs standard cloud egress fees, which can be significant for large datasets. Databricks supports Cloudflare R2 as a Delta Sharing storage backend (R2 charges zero egress fees) as a workaround for cross-cloud sharing egress.

PrivateLink cost at scale

At 50 VPCs each needing three Interface Endpoints to Databricks (workspace, SCC, REST), and each in 3 AZs: 50 × 3 × 3 = 450 ENIs at $0.01/hr = $3,240/month, before data transfer. The Databricks PrivateLink model uses one endpoint per workspace per VPC (not per AZ — the NLB handles AZ distribution), so the math is more like 50 × 3 = 150 ENIs = $1,080/month. Still significant; plan it into the platform cost model.

9Architect's take

The single most important networking habit for data platforms: deploy a Gateway Endpoint for S3 on day one. It is free, requires one Terraform resource, and eliminates a class of accidental-egress-over-NAT costs that are invisible until the AWS bill arrives. Every private subnet in every VPC that runs query engines, data pipelines, or Spark clusters should have a Gateway Endpoint route.

Default to PrivateLink for data platform services, not VPC Peering. Query engines, catalogs, and control planes expose specific API surfaces — they don't need full bilateral network access. PrivateLink gives you the minimum surface area: one service, one TCP port, no blast radius into the wider VPC. The cost is higher than peering (per-ENI hourly) but the security posture and operational simplicity are worth it. Peering is correct only when you genuinely need broad network access, e.g., a compute cluster that needs to reach many different ports across a shared services VPC.

Use Delta Sharing as the default pattern for any cross-org data access. Before building a PrivateLink or VPN between your org and a partner's, ask whether Delta Sharing can solve it instead. For read-only tabular data sharing, it almost certainly can — and you eliminate the permanent operational coupling between two organizations' network teams. The protocol is now a VLDB-published standard with implementations across Databricks, Snowflake, Apache Spark, Power BI, and pandas.

Watch for three developments over the next 12–18 months: (1) The Delta Sharing Network Gateway (announced Summer 2025) maturing from preview — once stable, it closes the "private storage" gap that still requires NCC+PrivateLink configurations for fully air-gapped providers. (2) AWS Verified Access expanding to cover more access patterns, potentially reducing the need for full VPN for developer/admin access to private resources. (3) Increasing PrivateLink adoption for IRC catalog endpoints — as Polaris, Lakekeeper, and Unity Catalog all expose REST APIs, expect all major platforms to offer Interface Endpoints for catalog access, making L3 as network-isolated as L0 storage already is.

Decision rule of thumb

When you hear "we need to share data with another account/org," answer with a question: "do they need network-level access (to deploy their own compute into our environment) or data-level access (to read specific tables)?" Network-level: VPC Peering or TGW. Data-level: Delta Sharing. Only about 10% of cross-org data sharing scenarios genuinely require network-level access. Delta Sharing handles the other 90% with less operational complexity, better governance, and no bilateral network trust.

Glossary

CIDR (Classless Inter-Domain Routing)
IP address range notation, e.g., 10.0.0.0/16 = 65,536 addresses. VPCs are defined by CIDR blocks; subnets are sub-ranges within them.
ENI (Elastic Network Interface)
A virtual NIC in AWS, attached to an EC2 instance or a service endpoint. Has one primary private IP, optional secondary IPs, and an associated security group.
NLB (Network Load Balancer)
An AWS L4 (TCP/UDP/TLS) load balancer. Required as the backend for a PrivateLink VPC Endpoint Service; handles millions of connections per second at very low latency.
NCC (Network Connectivity Configuration)
A Databricks object that governs how serverless compute connects to external resources. Can include PrivateLink rules that map a hostname to a VPC Endpoint Service, routing serverless traffic through a private path rather than the internet.
Pre-signed URL
A time-limited URL for an object in cloud storage that embeds authentication credentials in the query string. Confers the rights of the signing identity for the object, for the duration specified (typically 15 minutes in Delta Sharing).
STS Token (Security Token Service)
Temporary AWS credentials (AccessKeyId + SecretAccessKey + SessionToken) with a bounded expiry and optional policy constraints. Delta Sharing uses prefix-scoped STS tokens as an alternative to per-file pre-signed URLs — more efficient for large tables.
Transitive routing
Traffic flowing A→B→C through a network hub. VPC Peering explicitly prohibits it; Transit Gateway enables it by design.
VPC Endpoint Service
The provider-side construct for AWS PrivateLink. Backed by a Network Load Balancer; assigned a service name (com.amazonaws.vpce.region.vpce-svc-xxx). Consumers reference this name when creating Interface Endpoints.
Andromeda (GCP)
Google Cloud's software-defined networking fabric, implementing Virtual Private Cloud networking and Private Service Connect. Fully distributed; no centralized network appliances in the data path.
Delta Sharing Network Gateway
A Databricks-managed HTTPS proxy (announced Summer 2025, in preview) that serves as an intermediary for Delta Sharing traffic when the provider's storage is fully private. Recipients route their data reads through the gateway, eliminating the need for direct storage access.
