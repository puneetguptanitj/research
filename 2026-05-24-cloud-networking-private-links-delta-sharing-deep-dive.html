Data Platform Deep-Dive

Cloud Networking for Data Platforms

The full taxonomy of cloud network isolation and connectivity primitives — VPC peering, private links, transit gateways, NAT gateways, and how Databricks Delta Sharing collapses the cross-org networking problem to plain HTTPS.

Prepared 2026-05-24 Reading time ~30 min Focus Architecture · Mechanisms · Punch-hole patterns · Delta Sharing Clouds AWS · Azure · GCP (AWS primary)

TL;DR

Cloud VPC networking gives you a private, isolated network slice. The primitives you use to connect that slice to the world — or to other slices — each trade off scope, security, cost, and operational burden in distinct ways. Choosing the wrong one is how platforms end up with either an over-exposed attack surface or an $80k/year transit gateway bill for shuffling internal data.

Delta Sharing is architecturally interesting because it can sidestep the cross-account/cross-cloud networking problem — but only under a specific storage posture. Instead of requiring a PrivateLink or VPN between two organizations, it vends short-lived, scope-limited pre-signed URLs or STS tokens over HTTPS. Pre-signed URLs are not the same as making a bucket public: they are SigV4-signed, time-limited credential delegations that work even with S3 Block Public Access fully enabled. However, if the provider's bucket policy includes an aws:SourceVpce condition — restricting access to traffic that arrives through a specific VPC Endpoint — then pre-signed URL requests arriving from the recipient's internet egress IP are denied outright, regardless of signing. This is the correct posture for a production data lakehouse, and it means Delta Sharing does not eliminate networking work in that case. Databricks addresses this with the Delta Sharing Network Gateway (a proxy in the provider's VPC that fetches from the private bucket and streams to recipients over HTTPS) and NCC + PrivateLink (private connectivity between workspaces). The accurate framing: Delta Sharing sidesteps cross-org networking when storage is internet-reachable; for VPC-endpoint-only buckets, you still need a gateway proxy or PrivateLink — just at a higher layer than raw VPC peering.

  1. Internet Gateway ≠ NAT Gateway ≠ VPN Gateway ≠ Transit Gateway. Each serves a different traffic direction and scope. Conflating them is the root of most network architecture mistakes.
  2. PrivateLink exposes a specific service, not a network. This is the critical distinction from VPC Peering: no transitive routing, no broad blast radius, works across overlapping CIDRs. Use it for SaaS patterns and data platform services (Databricks, S3 Interface Endpoint, Redshift).
  3. VPC Peering is fine for two VPCs; Transit Gateway is the answer beyond that. The N*(N-1)/2 peering math, combined with the 125-connection limit and no transitive routing, makes peering unworkable at mesh scale.
  4. "Punching a hole" has a right and a wrong way. The right way is PrivateLink or a tightly-scoped security group rule with your own service behind an NLB. The wrong way is opening a CIDR range in a NACL without a compensating security group, or enabling full VPC peering when you only need service-level access.
  5. Delta Sharing reduces — but does not always eliminate — cross-org networking work. The protocol vends pre-signed URLs / STS tokens scoped to exactly the queried files. This works over the internet when storage is internet-reachable (Block Public Access ≠ no pre-signed URLs). But if the provider's bucket policy restricts access via aws:SourceVpce — the right posture for a production lakehouse — internet-originating pre-signed requests are denied. In that case you need a Delta Sharing Network Gateway (provider-side proxy) or NCC + PrivateLink between workspaces. Delta Sharing still eliminates the need for a bilateral network relationship; it does not eliminate the need to think about storage reachability.
  6. Gateway VPC Endpoints for S3 are free and should always be deployed in any private subnet that reads from S3. Using NAT Gateway for S3 traffic instead costs roughly $0.045/GB that could be zero.

1What it is & core architecture

A Virtual Private Cloud (VPC) is a logically isolated segment of a cloud provider's network, defined by one or more CIDR blocks (e.g., 10.0.0.0/16) that you own. Inside it, you create subnets — slices of that CIDR bound to a specific availability zone. Subnets are either public (route table has a default route to an Internet Gateway) or private (no IGW route; egress via NAT or none). Route tables, security groups (stateful L4 firewall per ENI), and NACLs (stateless L3/L4 firewall per subnet) control traffic within and across the VPC boundary.

The primitives that connect VPCs to other networks, or expose services between them, fall into six families:

The six families — detailed

① Internet Gateway (IGW)

What makes a subnet "public"

There is no "public subnet" flag in AWS. A subnet is public if — and only if — its route table has a 0.0.0.0/0 → igw-xxx entry. Delete that entry and the subnet instantly becomes private. The subnet itself has no property that changes. This is the entire distinction: public subnets route default traffic to an IGW; private subnets route it to a NAT Gateway or nowhere.

The public IP does not live on the instance

This is the most common mental model mistake. If you SSH into an EC2 and run ip addr or ifconfig, you will only see the private IP (e.g., 10.0.1.5). The public IP (e.g., 54.23.1.10) exists only in AWS's mapping table — it is never configured on the network interface of the instance. The IGW holds this mapping and applies it to every packet crossing the VPC boundary.

This has a critical implication: even if an instance is in a public subnet (route table has IGW entry), if it has no public IP assigned — either no auto-assign at launch or no Elastic IP attached — the IGW has nothing to map. The instance is invisible to the internet regardless of route tables or security groups.

What the IGW actually does

The IGW performs 1:1 static NAT in both directions. Inbound: a packet arrives destined for 54.23.1.10; the IGW rewrites the destination to 10.0.1.5 and forwards it into the VPC. Outbound: a packet leaves with source 10.0.1.5; the IGW rewrites the source to 54.23.1.10 before it exits to the internet. Without this translation, the internet would see an RFC-1918 private address as the packet source and have no route back.

The IGW has no IP address of its own, is not a hop in a traceroute, and has no single-device implementation. It is a distributed mapping function — address translation logic embedded in the AWS Hyperplane fabric running across every Nitro hypervisor host in the region. "Internet Gateway" is a conceptual name for a VPC boundary, not a description of a physical appliance. One VPC can have at most one IGW attached.

Key insight

The IGW is really a distributed NAT translation service that marks the logical boundary between your VPC's private address space and the public internet. It would be more technically accurate to call it a "VPC NAT boundary service." The name "gateway" is retained because it is easy to reason about on architecture diagrams — it represents the concept of a boundary crossing even though physically it is metadata that influences AWS Hyperplane behavior.

How internet routers know about the public IP — BGP

The IGW has nothing to do with how the internet routes traffic to AWS. That is handled entirely by BGP (Border Gateway Protocol) at a completely different layer — AWS's edge routers that peer with ISPs and internet exchanges (Equinix, DE-CIX, etc.). AWS owns large blocks of public IP ranges (publicly listed at ip-ranges.amazonaws.com/ip-ranges.json) and announces these prefixes to the global internet via BGP from its edge routers.

Internet routers do not have a specific route for your instance's IP like 54.23.1.10/32. They hold a route for the aggregated block — something like 54.0.0.0/8 → AWS. This prefix aggregation keeps the global routing table manageable. Once the packet reaches AWS's network edge, AWS's internal routing takes over and delivers it to the right VPC and IGW. The IGW is never involved in BGP and never speaks to the outside internet directly — by the time a packet reaches the IGW, it has already traversed the entire global routing system and arrived inside AWS's network.

Why a public-subnet EC2 can still be unreachable

Having an IGW route + public IP is necessary but not sufficient. In rough order of how often each blocks traffic:

  1. Security Group inbound rules — the most common cause. No rule allowing the port (22, 443, etc.) means the packet is dropped before reaching the OS. Security groups are stateful, so only inbound rules matter for externally-initiated connections.
  2. Network ACLs (NACLs) — subnet-level, stateless. A NACL must allow inbound on the destination port and outbound on the ephemeral return port range (1024–65535). Forgetting the outbound ephemeral rule silently kills connections.
  3. No public IP on the instance — launched without auto-assign public IP and no Elastic IP attached. The IGW has no mapping to apply.
  4. OS-level firewall — iptables or ufw blocking the port regardless of AWS-level controls.
  5. Service not listening — the connection reaches the OS but nothing is bound to that port.
  6. Instance not running / crashed — no response at the IP/port.

Debug order: route table → security group inbound → NACL in+out → public IP exists → OS firewall → service listening. Security groups catch ~90% of cases.

② NAT Gateway

What the NAT Gateway actually does — SNAT mechanics

A NAT Gateway performs SNAT (Source Network Address Translation): when a private-subnet instance sends a packet to the internet, the NAT GW rewrites the packet's source address from the instance's private IP (e.g., 10.0.2.10) to its own Elastic IP (e.g., 54.10.0.1) before handing the packet to the IGW. Simultaneously it records a connection tracking entry — the 5-tuple of protocol, source IP, source port, destination IP, destination port — so it knows how to reverse-translate the response when it arrives. When the remote server replies to 54.10.0.1:ephemeral-port, the NAT GW looks up the tracking table, rewrites the destination back to 10.0.2.10:original-port, and delivers it to the instance. The instance never sees anything but private addresses; the public EIP exists only on the NAT GW itself.

This is many:1 SNAT, not 1:1 NAT like the IGW. Hundreds of instances in the private subnet can share a single NAT GW and a single EIP, each identified by a unique source-port in the connection table. The NAT GW supports up to 55,000 simultaneous connections per unique destination (IP + port + protocol), and up to 900 new connections per second to the same destination — beyond that, connection errors appear. AWS documentation calls this "port allocation exhaustion" and it is a real failure mode on high-fan-out workloads (e.g., a Lambda function invoking hundreds of external APIs per second).

The public IP lives on the NAT Gateway, not on your instance

Unlike the IGW case — where the instance itself holds an EIP and the IGW maps it — with a NAT Gateway the instance has no public IP whatsoever. Running ip addr inside the instance shows only its private RFC-1918 address. The EIP (54.10.0.1) is allocated to and owned by the NAT GW resource. This has an important implication: you cannot allowlist individual private-subnet instance IPs at an external firewall — you can only allowlist the NAT GW's EIP, which all instances in that subnet share. This is intentional (and a security feature), but it means IP-based allowlisting at a partner or SaaS service is tied to your NAT GW EIP, not any particular workload.

Why the NAT Gateway must live in a public subnet

This is the question that trips people up most often. A NAT Gateway needs to forward translated packets to the internet. For that to work, the subnet it lives in must have a route to an Internet Gateway (0.0.0.0/0 → igw-xxx). A private subnet has no such route — traffic sent to 0.0.0.0/0 either hits another NAT GW or goes nowhere. If you placed a public NAT GW in a private subnet, the forwarded packet would have nowhere to go after translation. The NAT GW's own subnet's route table is what gives outbound packets their path to the internet — so that subnet must be public by definition.

Said differently: the NAT GW sits at the boundary between the private subnet (no IGW route) and the public subnet (has IGW route). Private instances route their internet-bound traffic to the NAT GW. The NAT GW translates and re-injects the packet into the public subnet, where the IGW then performs its own 1:1 mapping from the NAT GW's EIP to itself and sends the packet out. There are two hops for outbound traffic from a private instance: private subnet → NAT GW (SNAT) → public subnet → IGW (EIP mapping) → internet.

Key insight — NAT Gateway is not an ASG of EC2s

A tempting mental model: "NAT Gateway is just a fleet of EC2 instances auto-scaling behind a load balancer to do NAT." This is close enough to reason about why it scales but breaks down when you reason about failure modes or operational properties. NAT Gateway is a managed network appliance implemented in AWS Hyperplane — a fleet of Nitro-based bare-metal servers running stateful NAT tracking logic in the hypervisor/networking layer, not in userspace on EC2. There is no OS you could SSH into, no security group on the NAT GW itself (only NACLs apply to the subnet), no AMI, and no instance type choice. It is purpose-built, firmware-level translation logic.

The practical consequences: (1) No single-instance bottleneck — the Hyperplane fleet distributes connection tracking horizontally, so there is no "hot node" failure within an AZ. (2) Zonal, not regional — unlike an ASG that can span AZs, each NAT GW is strictly single-AZ. If the AZ fails, that NAT GW is gone with no automatic failover. (3) Scales to 100 Gbps automatically — you never provision capacity or choose an instance size. (4) Cost at idle — you pay $0.045/hr just for the NAT GW existing, which an ASG of stopped instances would not charge you for. This is the "always-on network appliance" cost model, not the "pay for compute you run" model.

How the Hyperplane fleet shares one EIP — distributed connection tracking

The EIP (54.10.0.1) is a logical routing label, not a property of any single physical host. AWS's internal fabric is configured to deliver all packets destined for that EIP to this AZ's Hyperplane fleet — analogous to anycast, but entirely within AWS's network. The fleet can have many Nitro bare-metal nodes, and any of them can handle any connection because they all share a distributed connection tracking state. This is the key mechanism: the 5-tuple mapping (EIP:source-port ↔ private-IP:original-port) is replicated across nodes, not stored locally on whichever node first handled the connection. So if Node 1 fails mid-connection, Node 2 already has the tracking entry and can serve the response packet without any disruption to the instance.

The EIP's uniqueness guarantee is maintained through ephemeral source port allocation. Because every instance's outbound connection is NAT'd to the same EIP but a distinct source port, the full 5-tuple (protocol + EIP + source-port + destination-IP + destination-port) remains unique per connection. The internet sees one address; the fleet's connection table turns that address + port into the correct private instance. This is also why the limit is per destination — two different EC2 instances connecting to the same external server on the same port can exhaust the ~55,000 available ephemeral ports for that destination before they exhaust any other resource.

Public NAT vs Private NAT Gateway

Public NAT Gateway is the default and common case: it lives in a public subnet, holds an EIP, and is used to give private-subnet resources outbound internet access. All traffic it forwards exits to the internet via the subnet's IGW.

Private NAT Gateway has no EIP and never sends traffic to the internet. Its purpose is entirely different: resolving overlapping CIDR problems between VPCs connected via Transit Gateway or VPC Peering. Suppose VPC-A (10.0.0.0/16) and VPC-B (10.0.0.0/16) need to communicate — their address spaces overlap, so standard routing cannot differentiate them. A Private NAT GW in one VPC translates the source IP of outgoing packets to an address in a non-overlapping range before they traverse the peering or TGW attachment. The receiving VPC sees a translated source IP and can route responses back. This is a niche but important use case for legacy multi-VPC environments or post-merger integrations where CIDR re-numbering is not feasible.

Zonal design and the HA pattern

A NAT Gateway is a zonal resource: one NAT GW lives in exactly one Availability Zone and cannot span AZs. AWS guarantees high availability within the AZ (via Hyperplane fleet redundancy) but provides zero protection against AZ failure. The correct HA pattern is: one NAT GW per AZ, and each private subnet's route table points to the NAT GW in the same AZ. If AZ-a fails and takes its NAT GW down, only AZ-a private subnets lose internet egress — AZ-b and AZ-c are unaffected because they each have their own NAT GW.

A common mistake is deploying a single NAT GW in one AZ and routing all private subnets — including those in other AZs — to it. This saves ~$0.045/hr per skipped NAT GW but creates a single point of failure and adds cross-AZ data transfer charges ($0.01/GB in us-east-1). On a data platform with significant S3 or API traffic, the cross-AZ transfer cost can easily exceed the NAT GW hourly fee within days.

Why a private-subnet instance can still fail to reach the internet

Having a route to a NAT GW is necessary but not sufficient. Failure order by frequency:

  1. No 0.0.0.0/0 → nat-gw-xxx route in the private subnet's route table — the most common cause. The NAT GW exists but the subnet was never wired to it. The instance sends internet-bound packets to the local VPC router, which has no route and drops them silently.
  2. Security group outbound rules — security groups on the instance deny outbound traffic on the target port. Default SGs allow all outbound; custom SGs often restrict it.
  3. NACL outbound + inbound rules — stateless, so both directions must be open. Must allow outbound on the destination port and inbound on ephemeral return ports (1024–65535). Forgetting ephemeral inbound silently drops all responses.
  4. NAT GW is in wrong AZ — the route table points to a NAT GW in a different AZ. Traffic flows but incurs cross-AZ charges, and if the remote AZ fails, so does this instance's egress.
  5. NAT GW subnet has no IGW route — the NAT GW was created in a private subnet (misplaced). It exists but can't forward packets anywhere. The NAT GW will show as "available" in the console but traffic silently fails.
  6. EIP not associated — rare, but if the EIP was disassociated after NAT GW creation, outbound packets have no public source address and the IGW rejects them.
  7. Connection tracking exhaustion — high-concurrency workloads hitting the 55,000 simultaneous connections or 900 new connections/sec per destination limit. New connections fail with timeout; existing ones continue normally.

Debug order: route table on private subnet → NAT GW subnet has IGW route → security group outbound → NACL outbound+inbound-ephemeral → NAT GW state is "available" → CloudWatch NAT GW error metrics (ErrorPortAllocation, PacketsDropCount).

Cost traps

S3 via NAT Gateway: $0.045/hr + $0.045/GB (us-east-1). Every gigabyte of S3 traffic routed through a NAT GW costs $0.045 that a free Gateway VPC Endpoint would eliminate. On a data platform moving 10 TB/day to S3 that is ~$450/day of avoidable cost. Gateway Endpoints for S3 and DynamoDB are free and should always be deployed in any private subnet that accesses those services — you add a route-table entry pointing S3's prefix list to the endpoint and traffic bypasses the NAT GW entirely.

Cross-AZ NAT: Routing private subnets in AZ-b through a NAT GW in AZ-a incurs $0.01/GB cross-AZ data transfer on top of the normal $0.045/GB NAT processing fee. At scale this is a meaningful cost; always co-locate NAT GW and private subnets in the same AZ.

Idle cost: Unlike EC2, a NAT GW charges $0.045/hr even when zero bytes pass through it. In dev/test environments with many isolated VPCs, idle NAT GWs can become a significant line item. The pattern here is to use a single shared NAT GW per AZ across multiple non-prod VPCs via Transit Gateway, rather than one NAT GW per VPC.

③ VPN Gateway & Direct Connect Gateway

Both primitives address hybrid cloud connectivity — linking an on-premises data center or corporate network to one or more VPCs. They differ radically in provisioning time, cost, bandwidth, and latency predictability. A Virtual Private Gateway (VGW) terminates IPSec/IKEv2 Site-to-Site VPN tunnels. Each VPN connection has two tunnels for redundancy (active/passive or active/active with ECMP); each tunnel supports up to ~1.25 Gbps. VGW integrates directly with on-premises hardware (Cisco, Juniper, Palo Alto) or software VPN appliances. Provisioning takes minutes. A Direct Connect Gateway (DXGW) aggregates one or more dedicated fiber circuits to AWS (1, 10, or 100 Gbps) and can connect them to multiple VPCs across multiple regions — something a VGW cannot do. DX provides deterministic, sub-10 ms latency to the nearest DX location and is the right choice when you need consistent performance for database replication, low-latency trading, or bulk data egress. Provisioning a DX circuit takes weeks to months.

Key limitation: Neither VGW nor DXGW solves cross-account or cross-org connectivity within AWS; they are strictly for reaching resources outside AWS. For multi-VPC routing combine with Transit Gateway (see ④).

④ Transit Gateway (TGW)

Transit Gateway is a regional hub router. VPCs, VPN connections, and Direct Connect attachments connect to it; TGW route tables define what can reach what. The core property is transitive routing: if VPC-A and VPC-C are both attached to the same TGW, VPC-A can reach VPC-C without a direct peering connection between them. TGW also enables fine-grained segmentation: you create multiple route tables (e.g., "prod", "dev", "shared") and associate each attachment with exactly one of them. This replaces the N×(N−1)/2 full-mesh peering problem — ten VPCs need 45 peering connections in a mesh, but only 10 TGW attachments. Cross-region TGW peering links two regional TGWs, extending the hub-and-spoke backbone globally over the AWS backbone (no public internet). Cost: $0.05/attachment-hour + $0.02/GB (us-east-1).

AWS Hyperplane connection: Transit Gateway's data plane is built on AWS Hyperplane. Packets from an attached VPC's instance flow through the Nitro hypervisor into the Hyperplane fabric, are looked up against the TGW route table, and forwarded to the destination attachment — all within the AWS physical network with no intermediate router hop visible to the EC2. This is why TGW can deliver line-rate throughput per attachment with sub-millisecond added latency.

⚠ CIDR Constraint — Same as VPC Peering

All VPCs attached to a Transit Gateway must have non-overlapping CIDRs. The TGW route table maps destination prefix → attachment ID. If two attachments advertise the same CIDR (e.g., both using 10.0.0.0/16), the route table has an irreconcilable conflict — the router cannot decide which VPC a packet for 10.0.1.5 belongs to. AWS will reject the conflicting attachment or produce unpredictable routing. This is not a TGW-specific quirk; it is a fundamental property of IP routing at any layer.

Workaround: Private NAT Gateway in front of TGW. AWS introduced "private NAT Gateway" specifically for overlapping-CIDR scenarios. Place a private NAT GW in a dedicated transit subnet inside each VPC; it SNATs the source address to a non-overlapping range before traffic reaches the TGW. The TGW then sees distinct CIDRs and routes normally. The trade-off is the same as any SNAT: source IP is hidden, which breaks IP-based security group references and complicates audit trails. For new environments, the right fix is org-wide CIDR planning via AWS IPAM before VPCs are created — re-IPing a VPC later requires recreating subnets and updating every security group and route table that references an IP.

⑤ VPC Peering

A VPC peering connection creates a direct, bilateral routing relationship between two VPCs — same account, cross-account, or cross-region. AWS injects routes into both VPCs' route tables pointing at the peering connection ID; traffic flows directly between the two VPCs on the AWS backbone with no intermediate hop. Security groups in the destination VPC apply normally and can reference the peering connection's CIDR or use the peer's security group ID (same-region only). The hard constraints are: (1) no transitive routing — if A↔B and B↔C are peered, A cannot reach C through B; AWS explicitly blocks it; (2) CIDR overlap is forbidden — if 10.0.0.0/16 is used in both VPCs, the peering request is rejected; (3) the 125-connection limit per VPC. For a flat mesh of N VPCs you need N×(N−1)/2 connections — 10 VPCs need 45 peerings, 90 route table entries, and security groups referencing 9 peer CIDRs each.

When to use it: Two-VPC, stable topologies. Classic example: a "data plane" VPC peered with a "shared services" VPC (DNS, monitoring, authentication). Never design a full mesh beyond 4–5 VPCs; switch to Transit Gateway.

Gateway Endpoint mechanics

AWS maintains a managed prefix list for each supported service — a named set of IP CIDR ranges (e.g., pl-63a5400a for S3 in us-east-1) that it updates automatically. Creating a Gateway Endpoint injects pl-xxxxx → vpce-xxx into the route tables you specify. Any packet whose destination IP falls within the prefix list now skips the NAT GW entirely and goes directly over the AWS backbone. The application is unaware: s3.amazonaws.com still resolves to the same public IP — the route table is what changes the path. For same-region S3, traffic via NAT GW also stays within AWS's regional infrastructure; the Gateway Endpoint's benefit is purely eliminating the $0.045/GB NAT processing charge and enabling aws:SourceVpce enforcement — not a different physical network.

Same-region constraint

The prefix list contains S3 IPs for the same region only. An EC2 in us-east-1 accessing s3.eu-west-1.amazonaws.com will not match the us-east-1 prefix list — the route falls through to 0.0.0.0/0 → nat-gw, incurring the $0.045/GB charge even with a Gateway Endpoint deployed. Silent cost trap on multi-region data platforms. Cross-region options: (a) native cross-region Interface Endpoint (launched Nov 2025 — no TGW/peering needed; supported services: S3, IAM, ECR, KMS, ECS, Lambda, Firehose, Route 53; requires vpce:AllowMultiRegion IAM permission; adds inter-region data transfer charges); (b) Interface Endpoint in the remote region's VPC via TGW/peering (still valid for all services); (c) NAT GW default route; or (d) S3 Multi-Region Access Points (MRAPs) + Interface Endpoint.

How pl-xxxx actually works at the data plane

The pl-63a5400a you see in the route table is a management-plane name for a container of IP CIDRs — for S3 in us-east-1 that is currently 40+ ranges: 52.216.0.0/15, 54.231.0.0/17, 3.5.0.0/19, 52.92.0.0/17, and dozens more. The VPC router's data plane expands the prefix list to its constituent CIDRs and performs normal longest-prefix-match against all of them. At the packet-forwarding level, it is indistinguishable from having 40+ individual CIDR rows all pointing at the same next-hop. The pl-xxxx name exists purely so AWS can silently update those CIDRs when S3 adds new IP blocks — your route table entry never changes. Inspect the current CIDRs any time: aws ec2 get-managed-prefix-list-entries --prefix-list-id pl-63a5400a

What vpce-xxx actually is — a Nitro-level forwarding token, not an IP

vpce-0abc1234 is not an IP address, not an ENI, and not a physical device. It is a logical forwarding token that the Nitro hypervisor on the sending EC2 host recognises as a special instruction. When a packet's destination IP matches one of the prefix-list CIDRs and the next-hop is a gateway endpoint, the Nitro hypervisor does three things: it does not forward the packet toward the Internet Gateway or NAT Gateway; it routes the packet directly to S3's fleet over AWS's internal network fabric; and it tags the request with the endpoint ID as metadata. That tag is what populates aws:SourceVpce on the S3 side — S3 receives the HTTP request already annotated with the vpce identity, which is how bucket policies enforce VPC-only access. Without this tag, S3 has no way to distinguish the request from one arriving over a public EIP.

Key mechanic

A Gateway Endpoint route entry is really two things simultaneously: a data-plane CIDR expansion (the managed prefix list expands to individual CIDRs at forwarding time) and a Nitro-level forwarding policy (the vpce-xxx token is not a network hop, it is an instruction to the hypervisor). Neither is a traditional IP next-hop.

The aws:SourceVpce condition — enforcing VPC-only bucket access

Add "Condition": {"StringNotEquals": {"aws:SourceVpce": "vpce-xxxx"}} to a bucket Deny statement and the bucket rejects any request that does not arrive through that specific endpoint — regardless of IAM identity. aws:SourceVpce evaluates the network path, not the credentials. This is the correct production posture: a stolen IAM key cannot exfiltrate data from outside the authorized VPC.

This is also why Delta Sharing's pre-signed URL approach breaks when aws:SourceVpce is present. A pre-signed URL carries valid SigV4 credentials but encodes no routing information — a recipient hitting the URL from their internet egress IP is denied on the path condition. The Delta Sharing Network Gateway resolves this by fetching from S3 inside the provider's VPC (satisfying the endpoint condition) and proxying the bytes to the recipient over HTTPS.

Why Gateway Endpoints only exist for S3 and DynamoDB

Route-table injection requires a service to be reachable at a stable, published, non-overlapping IP range — something only S3 and DynamoDB have as AWS-managed prefix lists. Most other services (KMS, Secrets Manager, ECR, STS, etc.) share IP space or change IPs too frequently. More fundamentally, prefix-list redirection works without DNS override, which is only safe when the service hostname is irrelevant to the path — for everything else, the application must connect to a private IP, which requires DNS override, which requires an ENI. That means Interface Endpoint.

The deciding factor is whether the service's IP ranges are stable and enumerable. S3 and DynamoDB have always had their ranges published in AWS's ip-ranges.json. Services like SQS, KMS, Secrets Manager, and SSM are accessed by DNS hostname; their backend IPs are opaque, not published, and scale dynamically as capacity adjusts. There is no prefix list to build, and therefore no route table entry to write.

Historical context

Gateway Endpoints (2017) predate Interface Endpoints / PrivateLink (2018). They were AWS's first private-access mechanism. When PrivateLink launched as a generalised solution for any service, AWS kept Gateway Endpoints for S3 and DynamoDB because they work, are completely free (no ENI hourly charge, no data-processing fee), and require no per-AZ planning. They are effectively a legacy mechanism with no reason to deprecate — but PrivateLink is the right mental model for everything else.

Consumer-side mechanics are topology-invariant

The Interface Endpoint picture above is identical whether the provider is in the same account, a different account, or (as of Nov 2025) a different region. Your VPC always gets an ENI with a private IP; your DNS resolver always returns that private IP; your EC2 always sends traffic to a 10.x.x.x address — it never sees the provider's topology. What varies: cross-account adds an allowlist/acceptance step on the provider's Endpoint Service and you reference its service name when creating your endpoint. Cross-region (Nov 2025, select AWS services): you specify the remote service's region at endpoint creation time (--service-region us-west-2); AWS still creates the ENI in your local subnet as normal. Two things differ from same-region: DNS gives you only a regional record (no per-AZ zonal record), and you pay the standard EC2 inter-region data transfer rate on top of normal PrivateLink pricing. The ENI + DNS override + Hyperplane abstraction holds in all three cases; only the authorization config and pricing change.

What the Interface Endpoint actually does — ENI creation and DNS override

When you create an Interface Endpoint and specify subnets (one per AZ), AWS creates one ENI per subnet. Each ENI receives a private IP from that subnet's CIDR — e.g., 10.0.3.47. These are visible in the EC2 console as "interface endpoint ENIs." Simultaneously, if Enable Private DNS Names is selected (requires enableDnsHostnames and enableDnsSupport both set to true on the VPC), AWS creates a private hosted zone that overrides the service's public DNS name. For example, creating an Interface Endpoint for S3 overrides s3.us-east-1.amazonaws.com so the VPC Resolver returns 10.0.3.47 instead of S3's public IP. The application does not change — it still calls the same hostname — but inside the VPC the resolver returns the ENI's private IP. Traffic to 10.0.3.47 goes to the endpoint, never reaching the internet.

Unlike a Gateway Endpoint, the Interface Endpoint ENI has a security group. You must manage inbound rules on it: by default the auto-created security group allows all traffic, but in practice you should restrict inbound to port 443 (or the service port) from the consumer's CIDR or security group. The EC2's outbound rules must also permit traffic to the ENI. This is the single most common Interface Endpoint failure mode — DNS resolves correctly to the ENI IP, but the ENI's security group blocks the connection, and the application sees a timeout indistinguishable from a routing failure.

How PrivateLink traverses Hyperplane — no centralized proxy in the path

The endpoint ENI is not a standard Nitro-card-backed network interface on a specific physical host. It is an endpoint ENI — a logical handle backed by the Hyperplane fleet. When traffic from your EC2's Nitro hypervisor is destined for the ENI's private IP, the hypervisor recognizes the destination as a PrivateLink endpoint and intercepts the packet, injecting it directly into the Hyperplane inter-host fabric rather than routing it through the subnet's standard path. Hyperplane performs a distributed DNAT at the source host: the packet's destination is rewritten from the endpoint ENI IP to the actual NLB target IP in the provider VPC, and the packet is forwarded across the physical datacenter fabric. The source IP is simultaneously translated to a Hyperplane-managed address in the provider's range, so the NLB sees a routable source for responses.

The consequence is that there is no single appliance or proxy in the PrivateLink path — no centralized "PrivateLink gateway" that all traffic must traverse. Each consumer EC2 host's Nitro hypervisor handles the translation independently. This is why PrivateLink has no per-endpoint throughput ceiling (it scales with the number of consumer hosts) and adds only microseconds of latency (one Hyperplane control-plane lookup, then direct physical-fabric forwarding). The availability of PrivateLink is identical to the availability of Hyperplane itself — the same fabric that powers NAT Gateway, Transit Gateway, and core VPC routing. If there is a Hyperplane event, PrivateLink, NAT GW, and TGW all degrade together.

Key insight — PrivateLink is a service connector, not a network connector

VPC Peering and Transit Gateway expose entire networks to each other — any ENI in VPC-A can potentially reach any ENI in VPC-B once connected. PrivateLink exposes a single service: specifically, whatever is behind one NLB. A consumer who creates an Interface Endpoint to your service cannot scan your VPC, cannot reach other services in your VPC, and cannot determine your VPC's CIDR. The blast radius is one service, not one network. This also means PrivateLink works with overlapping CIDRs — two VPCs both using 10.0.0.0/16 can use PrivateLink between them with no conflicts, because there is no routing relationship between the VPCs at all. This is the architectural distinction that makes PrivateLink the correct pattern for SaaS exposure, multi-tenant data services, Databricks workspace connectivity, and cross-account resource access.

The provider side — NLB, endpoint service, and acceptance workflow

Exposing a service via PrivateLink requires three things. First, the service must run behind a Network Load Balancer — not an ALB. NLB is required because PrivateLink operates at L4; Hyperplane hands off a TCP/UDP connection to the NLB and relies on the NLB preserving the 5-tuple for connection tracking. ALBs operate at L7 (HTTP/HTTPS) and do not interoperate with the Hyperplane PrivateLink path. Second, you create a VPC Endpoint Service resource referencing the NLB ARN. AWS assigns a service name: com.amazonaws.vpce.us-east-1.vpce-svc-xxx. Third, configure acceptance: by default, new endpoint connection requests require manual approval from the service owner. You can whitelist specific AWS account IDs or enable auto-accept (appropriate for internal org services, not for multi-tenant SaaS). Share the service name with consumers; they create Interface Endpoints referencing it.

One operational nuance: the Endpoint Service is zonal within the provider VPC. It serves traffic in the AZs where the NLB has healthy targets. A consumer ENI in AZ-a connects to the NLB target in AZ-a. If the provider has no healthy NLB target in AZ-b, consumer ENIs in AZ-b fail to connect. Always ensure the NLB has healthy targets in every AZ where consumers will create endpoint ENIs — this is the provider-side HA obligation.

Zonal design and the per-AZ ENI pattern

Interface Endpoints are VPC-scoped (one endpoint per VPC per service), but ENIs are subnet-scoped. You specify one subnet per AZ when creating the endpoint; AWS creates one ENI per specified subnet. The VPC Resolver returns the ENI IP in the same AZ as the caller — no cross-AZ traffic. If you only specify one AZ's subnet, all EC2s in other AZs still reach the endpoint ENI, but traffic crosses AZs and incurs the standard $0.01/GB cross-AZ charge on top of the endpoint's $0.01/GB processing fee. On a platform reading terabytes per day from a service, this silently doubles endpoint cost and adds a single-AZ failure dependency.

Cost model: $0.01/hr per AZ where an ENI exists + $0.01/GB data processed through the endpoint. Three AZs × $0.01/hr = ~$21.60/month for the endpoint to exist at idle. For S3 specifically, compare this to the Gateway Endpoint which is completely free. Use Interface Endpoints for S3 only when you need the DNS override (for client libraries that bypass prefix-list routing) or when your architecture specifically requires the private DNS behavior.

Why DNS alone cannot replace the ENI

A natural question: why not skip the ENI entirely and have the private DNS record resolve directly to the actual SQS/KMS backend IP? Fewer moving parts, no ENI cost, no Hyperplane indirection. It breaks on one hard structural constraint.

The SQS fleet, KMS nodes, and every other AWS managed service run on AWS's internal service network — a routing domain that is completely separate from your VPC. Your VPC has a CIDR (10.0.0.0/16) and a route table with entries for your VPC CIDR, peered VPCs, and attached gateways. It has no route to AWS's internal service network. Even if Route 53 handed your EC2 the actual IP of an SQS server, the packet would hit your VPC router, find no matching route, and get dropped. Knowing the destination IP is useless without a routable path.

The ENI solves exactly this: it lives in your subnet's CIDR — a VPC-routable address. Your router knows how to reach 10.0.3.47. From there, Hyperplane bridges the gap into the internal service network. Two additional reasons DNS-direct fails even if you punched a route through: (1) fleet churn — service backends appear and disappear on timescales of minutes; DNS TTLs of even 5s produce stale IPs and connection errors, while Hyperplane health-checks in real time; (2) security context lost — the aws:SourceVpce condition key is set by Hyperplane as it processes the request; bypassing Hyperplane means no endpoint identity tag is attached, so any resource policy gating on it silently denies.

Why Interface Endpoint failures look like DNS problems — debug order

  1. DNS resolves to a public IP, not the ENI IP — Enable Private DNS Names is not set, or the VPC has enableDnsHostnames or enableDnsSupport set to false. Fix: enable both VPC DNS attributes; update the endpoint to enable private DNS. Verify with nslookup service.region.amazonaws.com — an RFC-1918 result (10.x.x.x) means the override is working.
  2. Connection timeout after correct DNS — the endpoint ENI's security group blocks inbound on the service port (443 for most AWS services), or the EC2's outbound SG blocks the connection. Fix: allow inbound 443 on the endpoint SG from the EC2's CIDR or SG ID; allow outbound 443 from the EC2.
  3. HTTP 403 from the service — a aws:SourceVpce condition on the bucket/resource policy requires a specific endpoint ID, but the request came through a different endpoint or the internet. Fix: verify the endpoint ID in the bucket policy matches the endpoint in use, or remove the condition if cross-endpoint access is intentional.
  4. Endpoint ENI in wrong AZ — traffic succeeds but incurs cross-AZ cost — subnets for some AZs were not specified when creating the endpoint. Fix: modify the endpoint to add the missing AZ subnets.
  5. Endpoint state is "pending" or "failed" — the endpoint service requires manual acceptance and the connection has not been approved. Fix: approve the connection from the service owner's account in the VPC console under Endpoint Services → Endpoint connections.
  6. Service not available via PrivateLink in this region — not all AWS services support Interface Endpoints in every region. Check AWS documentation for the specific service. Verify the service name is correct; an invalid service name creates an endpoint that stays in "pending" indefinitely.

Cost traps

Interface Endpoint for S3 instead of Gateway Endpoint: Interface Endpoint for S3 costs $0.01/GB; the Gateway Endpoint for S3 is free. On a data platform moving 10 TB/day through S3, the difference is ~$100/day. Always use the Gateway Endpoint for S3 and DynamoDB. Reserve Interface Endpoints for S3 for cases where the DNS override is specifically required (e.g., path-style S3 access in a client library that bypasses prefix-list routing).

Idle per-AZ cost in large environments: Each AZ where an Interface Endpoint has an ENI costs $0.01/hr regardless of traffic. Three AZs × 10 Interface Endpoints (KMS, Secrets Manager, ECR, STS, SSM, etc.) = $0.30/hr = ~$216/month for endpoints to exist at idle. In dev/test VPCs with light traffic, this fixed cost can exceed the per-GB cost of routing the same services through a NAT GW. One pattern: deploy Interface Endpoints only in prod; dev/test workloads use NAT GW.

Single-AZ endpoint with multi-AZ workloads: Underspecifying subnets at endpoint creation means consumers in other AZs pay $0.01/GB cross-AZ transfer on top of $0.01/GB endpoint processing — silently doubling the effective endpoint cost for multi-AZ workloads. Always specify all AZ subnets at creation time.

Key distinction

VPC Peering and Transit Gateway expose entire networks to each other — any ENI in VPC A can potentially reach any ENI in VPC B (modulo security groups). PrivateLink exposes a single service. This is why PrivateLink is the correct pattern for SaaS exposure, multi-tenant data services, and cross-account access to specific resources: the blast radius is one service, not one network. Transit Gateway and VPC Peering are network-level connectors; PrivateLink is a service-level connector — and that distinction is the most important architectural decision in cloud network design.

Azure equivalents

Azure maps closely: Service Endpoints route traffic to Azure PaaS services over the Azure backbone but the service retains its public IP — effectively a policy-based route with no network isolation. Private Link / Private Endpoint injects the service into your VNet as a private IP (like an AWS Interface Endpoint) and requires a Private DNS Zone to override the public hostname. Azure Virtual Network Peering is the VPC Peering analog; Azure Virtual WAN is the Transit Gateway analog. One Azure nuance: VNet Service Endpoints are quick to set up and good enough for most intra-Azure scenarios, but they don't work cross-tenant; Private Endpoints do.

GCP equivalents

GCP's Private Service Connect (PSC) maps to PrivateLink. PSC is implemented via Google's Andromeda SDN — the data plane is fully distributed with no centralized proxy, and traffic is NAT'd at the destination host. A PSC endpoint creates a forwarding rule in the consumer VPC; the producer publishes a service via a load balancer backend. GCP also has VPC Service Controls, which is architecturally different from PSC: it creates an API-layer security perimeter around GCP services to prevent data exfiltration — complementary to PSC, not a substitute. VPC Network Peering is the GCP equivalent of VPC Peering, with the same no-transitive-routing constraint.

2Placement in the data platform & AI landscape

Cloud networking is not itself one of the L0–L9 data platform layers — it is infrastructure that underlies all of them. But it is far from neutral: every data platform design decision (where the catalog runs, whether the query engine can reach object storage without traversing the internet, how cross-org sharing works) is constrained or enabled by the networking primitives chosen. Think of cloud networking as the "L-1" plane — it determines what is reachable by L0 and above.

The practical intersections that matter for data platform architects:

  • L0 ↔ networking: How query engines reach S3/ADLS/GCS. Gateway Endpoints (S3) remove NAT cost. Interface Endpoints bring S3 fully private. VPC firewall rules on buckets enforce that only traffic from known CIDR/VPC Endpoint is accepted.
  • L3/L4 ↔ networking: Databricks workspace, Unity Catalog control plane, Trino coordinators — all should be reachable only via PrivateLink from consumer VPCs, not over public internet.
  • Cross-org sharing ↔ networking: The traditional model requires cross-account PrivateLink or VPC Peering + storage policy. Delta Sharing replaces this with an application-layer protocol.
  • L8 ↔ networking: Security groups are the first enforcement point. IAM roles control what an EC2/Lambda/container is allowed to do; security groups control what it can reach. Both are required.

3How each primitive works — mechanisms

VPC Peering — bilateral route injection

When you create a VPC Peering connection, AWS injects routes into the route tables of both VPCs pointing at the peering connection ID. Traffic flows directly between the two VPCs on the AWS backbone without any intermediate hop — no router appliance, no encryption overhead. The route table entries must be added manually (or via Terraform); they are not automatic. Security groups in the destination VPC apply normally; NACLs on both sides apply. Crucially, route tables are VPC-scoped: if VPC-B is peered with both VPC-A and VPC-C, there is no route in VPC-B's route table that lets A-originated traffic exit through the VPC-C peering — AWS explicitly prohibits transitive routing in peering. IP addressing must not overlap.

Transit Gateway — attachment-based hub routing

A TGW has one or more route tables. Each VPC, VPN, and Direct Connect attachment is associated with a route table and can propagate its CIDR to it. By default, a VPC attached to a TGW can reach all other attached VPCs — transitive routing works because the TGW is the transit router. You can create isolated route tables (one per VPC) to enforce fine-grained segmentation: a "prod" table that only accepts routes from prod attachments, a "dev" table that only accepts dev. Inter-region peering links two TGWs, extending the hub-and-spoke globally. TGW also terminates Site-to-Site VPN and integrates with Direct Connect Gateway for on-prem.

AWS PrivateLink — the hypervisor-layer shortcut

On the provider side: a service runs behind a Network Load Balancer. You create a VPC Endpoint Service referencing the NLB; AWS assigns it a service name. You optionally require acceptance of new endpoint connections (default: required for security — do not disable this). On the consumer side: you create an Interface VPC Endpoint specifying the service name; AWS creates one ENI per specified subnet, each with a private IP from that subnet's CIDR. AWS simultaneously creates a private DNS entry (a zone override) so that the service's public hostname (e.g., *.execute-api.us-east-1.amazonaws.com) resolves to the ENI private IP inside the VPC. Traffic from the consumer's EC2/Lambda goes to the ENI private IP; AWS rewrites it in the hypervisor and routes it directly to the NLB target — the public internet is never involved.

Key mechanism

PrivateLink's data plane is fully distributed (implemented in the Nitro hypervisor). There is no centralized proxy. Measured latency overhead vs. direct VPC access is typically under 100µs. The NLB in the provider VPC is not a bottleneck at typical enterprise data platform scale — it handles millions of connections per second.

Gateway VPC Endpoints — route table magic, zero cost

For S3 and DynamoDB only. When you create a Gateway Endpoint, AWS injects a "pl-XXXXXXXX" (prefix list) entry into the route tables you specify, routing all traffic to those service IP ranges through the gateway endpoint — bypassing NAT Gateway and internet. No ENI, no DNS change, no security group. Access is controlled by an endpoint policy (IAM-style JSON) and by the bucket policy (aws:SourceVpce condition key). The prefix list is managed by AWS and updated automatically as S3 IP ranges change.

How the DNS override works for PrivateLink

Watch out

Split-horizon DNS is a common failure mode. If "Enable Private DNS Name" is not set on the Interface Endpoint, or if the VPC does not have DNS resolution enabled, the application will resolve the service hostname to its public IP and either fail (if IGW is absent) or succeed over the internet (defeating the purpose). Always verify with dig <endpoint-hostname> from within the VPC — it must return a 10.x.x.x address, not a public IP.

4Punching holes in VPCs — when and how

"Punching a hole" is informal language for creating inbound access to a resource that would otherwise be unreachable from outside its subnet or VPC. The phrase covers a spectrum from a single security group rule change to full VPC peering. The decisions cascade: who needs access → what scope of network do they need → what is the minimum surface area.

10+ VPCs need to route to a shared service VPC Yes — scope to service PrivateLink from service VPC (avoids TGW cost) OR TGW centralized endpoint pattern

The right way to expose a service across VPCs

The canonical pattern is: service behind a Network Load Balancer → VPC Endpoint Service → consumers create Interface Endpoints. The NLB is required because PrivateLink's data plane connects to NLB targets, not to EC2 instances directly. The security group on the NLB target EC2s/containers should allow inbound only from the NLB security group (or NLB's CIDR), not from 0.0.0.0/0. This way the only path to the service is through the NLB → PrivateLink → consumer ENI.

Security Group vs NACL — which to touch

Always prefer security group rules over NACL rules for punching access holes. Security groups are stateful (return traffic is automatically allowed), attached per-ENI (so rules are scoped to exactly the affected resource), and support source-as-security-group-ID referencing (which eliminates IP management entirely for same-VPC resources). NACLs are stateless — you must add rules for both inbound and outbound traffic, and they apply to all traffic entering/leaving the subnet, not just the resource you care about. Use NACLs as a defense-in-depth second layer (e.g., "block all traffic from this subnet to the database subnet except on port 5432"), not as the primary access control.

Common mistake

Opening a NACL rule for 0.0.0.0/0 on a port "temporarily" and forgetting it. NACLs have no stateful tracking — a rule that's open is open to all traffic matching that pattern. Security groups with specific source security group IDs are both safer and easier to audit.

Zero-trust alternatives to "punching holes"

For connectivity patterns that don't fit neatly into VPC primitives — especially ephemeral developer access, third-party SaaS-to-private-VPC connections, or multi-cloud without a VPN — consider identity-aware proxies and mesh tools: AWS Verified Access (OIDC/SAML before reaching a resource, no VPN needed), Tailscale (WireGuard-based overlay mesh, peer-to-peer with DERP relay, works across clouds), or Cloudflare Tunnel (outbound-only connector from the private resource, no inbound firewall rules needed). These shift the trust model from "network location" to "identity + device posture" — better for humans and for service-to-service connections that cross trust domain boundaries.

5Authentication & identity

Cloud networking itself is not an authentication system — it controls reachability, not identity. But every networking primitive interacts with identity systems in specific ways, and confusing network-layer access control with application-layer authentication is a common security flaw.

PrivateLink: network approval ≠ authentication

When a consumer creates an Interface Endpoint to your service, PrivateLink requires the provider to accept the connection (or configure auto-accept). This is a network-level admission control — it answers "is this VPC Endpoint ID allowed to reach my NLB?" It does not authenticate the application calling the service. Any EC2 in the consumer VPC that can reach the Interface Endpoint ENI will get a TCP connection to the provider's NLB. The service itself must enforce authentication — typically via TLS mutual authentication (mTLS) or Bearer tokens. Treat PrivateLink acceptance as a coarse network gate, not a substitute for app-level authn.

IAM for Gateway Endpoints

Gateway Endpoints for S3 can carry an endpoint policy — an IAM resource policy attached to the endpoint that restricts what operations and which S3 buckets the VPC can access through it. Combined with a bucket policy using aws:SourceVpce, you get: (a) only traffic through the endpoint reaches the bucket, and (b) only the operations in the endpoint policy are permitted. This is the correct pattern for locking down data lake S3 buckets to only internal VPC access.

Databricks PrivateLink: workspace-level network controls

Databricks exposes its workspace, the SCC relay, and the REST API all via PrivateLink endpoints. The workspace URL resolves to the Interface Endpoint ENI in your VPC. Authentication to Databricks itself is separate: OAuth2 + OIDC (Databricks identity, or federated via your IdP), enforced at the Databricks control plane after the network-level connection is established. Unity Catalog enforces RBAC/ABAC on top of that. In other words, PrivateLink handles network reachability; Unity Catalog + Databricks authn handles identity and authorization. Neither replaces the other.

Subtle risk

Enabling PrivateLink for a Databricks workspace but leaving the workspace's public access enabled ("no-public-access = false") is not full isolation. Set no-public-access = true to ensure the workspace API is only reachable through the PrivateLink endpoint, not over the internet. This is the Databricks workspace-level analog of setting a bucket policy with aws:SourceVpce.

6Delta Sharing vs. private links — the architectural substitution

The traditional approach to cross-account or cross-org data sharing in the cloud is: (1) set up VPC Peering or PrivateLink between the two accounts, (2) configure storage bucket policies to allow the recipient's IAM role, (3) share IAM credentials or assume-role trust. This creates a permanent network-level relationship between two organizations — expensive to set up, fragile to maintain, and operationally coupled (provider network changes break recipient access).

Delta Sharing decouples data sharing from network topology. It is an open HTTP REST protocol (published on GitHub; VLDB 2025 paper: Puttaswamy et al.) that works as follows: the data provider runs a Delta Sharing server (Databricks' is built into Unity Catalog). The server maintains a catalog of shares, recipients, and access grants. When a recipient queries a share, the server returns pre-signed URLs (AWS S3 pre-signed URLs, Azure SAS tokens, or GCS signed URLs) or STS tokens that grant temporary, scoped, read-only access to exactly the Parquet/Delta files needed for that query. The recipient reads directly from object storage using those credentials — no permanent network path to the provider's VPC is required.

How Delta Sharing vends credentials — the credential flow

When Delta Sharing does NOT eliminate network configuration

Delta Sharing simplifies networking for the common case (internet-accessible object storage). Three scenarios still require network configuration:

1. Provider storage has no public access. If the S3 bucket policy denies all public access and only allows the provider's VPC Endpoint, then pre-signed URLs generated by the Delta Sharing server will return HTTP 403 when the recipient tries to use them from outside the VPC. Solutions: (a) use a Delta Sharing Network Gateway (new in Summer 2025 — provider-side HTTPS proxy that re-serves the data, removing the need for recipients to reach the raw bucket), or (b) configure the bucket to allow traffic from the Delta Sharing server's IP ranges / VPC endpoint for the presign operation.

2. Fully private cross-cloud sharing (e.g., AWS recipient → Azure provider). If the provider's ADLS storage is private, the recipient's serverless compute needs a path. The pattern (Databricks, Jan 2025): configure an NCC (Network Connectivity Configuration) for serverless with a PrivateLink rule pointing at a VPC Endpoint Service in AWS, which routes via an NLB whose target group has the ADLS private endpoint IP. Cross-cloud: a Site-to-Site VPN or ExpressRoute/Direct Connect links AWS and Azure first. The NCC object tells Databricks serverless to use the PrivateLink path rather than the public internet for that hostname.

3. Recipient has strict egress controls. If the recipient's network blocks outbound internet but they need to reach the Databricks Delta Sharing endpoint, they need an Interface Endpoint for the Databricks workspace on the provider side, or they need their own Databricks workspace's NCC to route sharing traffic through a private path.

The key insight

Delta Sharing inverts the networking burden: instead of both organizations building a persistent bilateral network relationship (VPC Peering, PrivateLink, VPN), the protocol treats the internet as the transport and uses object storage's own pre-signed URL/STS mechanism as the access control. The networking problem collapses to "can the recipient make outbound HTTPS connections?" — which is almost always yes.

The N*(N-1)/2 peering problem, illustrated

With 10 VPCs in a full mesh, you need 45 peering connections, each requiring route table entries on both sides (90 entries), and security group rules referencing 9 peer CIDRs per VPC. If any two VPCs' CIDRs overlap (common in org mergers or multi-team accounts), the entire mesh breaks. Transit Gateway cuts this to 10 attachments, one route table, and works regardless of CIDR overlap with careful route table design.

Delta Sharing egress costs

Delta Sharing within a region incurs no egress cost from the cloud provider's perspective — the data files are accessed from the same region. Cross-region or cross-cloud sharing incurs standard cloud egress fees, which can be significant for large datasets. Databricks supports Cloudflare R2 as a Delta Sharing storage backend (R2 charges zero egress fees) as a workaround for cross-cloud sharing egress.

PrivateLink cost at scale

At 50 VPCs each needing three Interface Endpoints to Databricks (workspace, SCC, REST), and each in 3 AZs: 50 × 3 × 3 = 450 ENIs at $0.01/hr = $3,240/month, before data transfer. The Databricks PrivateLink model uses one endpoint per workspace per VPC (not per AZ — the NLB handles AZ distribution), so the math is more like 50 × 3 = 150 ENIs = $1,080/month. Still significant; plan it into the platform cost model.

9Architect's take

The single most important networking habit for data platforms: deploy a Gateway Endpoint for S3 on day one. It is free, requires one Terraform resource, and eliminates a class of accidental-egress-over-NAT costs that are invisible until the AWS bill arrives. Every private subnet in every VPC that runs query engines, data pipelines, or Spark clusters should have a Gateway Endpoint route.

Default to PrivateLink for data platform services, not VPC Peering. Query engines, catalogs, and control planes expose specific API surfaces — they don't need full bilateral network access. PrivateLink gives you the minimum surface area: one service, one TCP port, no blast radius into the wider VPC. The cost is higher than peering (per-ENI hourly) but the security posture and operational simplicity are worth it. Peering is correct only when you genuinely need broad network access, e.g., a compute cluster that needs to reach many different ports across a shared services VPC.

Use Delta Sharing as the default pattern for any cross-org data access. Before building a PrivateLink or VPN between your org and a partner's, ask whether Delta Sharing can solve it instead. For read-only tabular data sharing, it almost certainly can — and you eliminate the permanent operational coupling between two organizations' network teams. The protocol is now a VLDB-published standard with implementations across Databricks, Snowflake, Apache Spark, Power BI, and pandas.

2025–2026 developments — what shipped and what's still in progress:

(1) Delta Sharing Network Gateway (announced June 2025, private preview) — still in preview as of mid-2026; not yet GA. Watch for GA: once stable it closes the "private storage" gap that still requires NCC+PrivateLink configurations for fully air-gapped providers.

(2) AWS Verified Access — ✓ shipped. Non-HTTP(S) support (SSH, RDP, and other TCP protocols) reached GA in mid-2025; FedRAMP High/Moderate authorization followed in March 2025. VPN-less secure access to RDS databases, EC2 over SSH/RDP is now production-ready. This meaningfully reduces the need for full Site-to-Site VPN for developer and admin access to private resources.

(3) Native cross-region PrivateLink for AWS services — ✓ shipped Nov 2025. S3, IAM, ECR, KMS, ECS, Lambda, Firehose, and Route 53 are now reachable cross-region over Interface Endpoints without TGW or VPC peering — one of the bigger 2025 networking shifts. Still watch: cross-region support for customer-owned VPCE services (non-Amazon), and Interface Endpoint adoption for IRC catalog endpoints (Polaris, Lakekeeper, Unity Catalog REST APIs).

Decision rule of thumb

When you hear "we need to share data with another account/org," answer with a question: "do they need network-level access (to deploy their own compute into our environment) or data-level access (to read specific tables)?" Network-level: VPC Peering or TGW. Data-level: Delta Sharing. Only about 10% of cross-org data sharing scenarios genuinely require network-level access. Delta Sharing handles the other 90% with less operational complexity, better governance, and no bilateral network trust.

Glossary

CIDR (Classless Inter-Domain Routing)
IP address range notation, e.g., 10.0.0.0/16 = 65,536 addresses. VPCs are defined by CIDR blocks; subnets are sub-ranges within them.
ENI (Elastic Network Interface)
A virtual NIC in AWS, attached to an EC2 instance or a service endpoint. Has one primary private IP, optional secondary IPs, and an associated security group.
NLB (Network Load Balancer)
An AWS L4 (TCP/UDP/TLS) load balancer. Required as the backend for a PrivateLink VPC Endpoint Service; handles millions of connections per second at very low latency.
NCC (Network Connectivity Configuration)
A Databricks object that governs how serverless compute connects to external resources. Can include PrivateLink rules that map a hostname to a VPC Endpoint Service, routing serverless traffic through a private path rather than the internet.
Pre-signed URL
A time-limited URL for an object in cloud storage that embeds authentication credentials in the query string. Confers the rights of the signing identity for the object, for the duration specified (typically 15 minutes in Delta Sharing).
STS Token (Security Token Service)
Temporary AWS credentials (AccessKeyId + SecretAccessKey + SessionToken) with a bounded expiry and optional policy constraints. Delta Sharing uses prefix-scoped STS tokens as an alternative to per-file pre-signed URLs — more efficient for large tables.
Transitive routing
Traffic flowing A→B→C through a network hub. VPC Peering explicitly prohibits it; Transit Gateway enables it by design.
VPC Endpoint Service
The provider-side construct for AWS PrivateLink. Backed by a Network Load Balancer; assigned a service name (com.amazonaws.vpce.region.vpce-svc-xxx). Consumers reference this name when creating Interface Endpoints.
Managed Prefix List (pl-xxxx)
An AWS-owned named container of IP CIDRs usable as a route-table destination or security group source. The VPC router expands it to constituent CIDRs at the data plane and performs normal longest-prefix-match. AWS updates the entries when service IP ranges change; references to the list require no maintenance.
Gateway Endpoint / vpce-xxx as a route target
A VPC endpoint type that injects a managed-prefix-list route into one or more route tables. The next-hop token (vpce-xxx) is a Nitro hypervisor–level forwarding instruction — not an IP address, not an ENI. Supported only for S3 and DynamoDB; free; no ENI, no per-AZ cost, no DNS override required.
Hyperplane
AWS's internal distributed NLB / network overlay, running on the same bare-metal substrate as Nitro cards. It implements Interface Endpoint ENIs as virtual entry points, performs health-checked distributed DNAT to service fleet backends, and bridges VPC routing domains into AWS's internal service network. Powers NAT Gateway, Transit Gateway, and PrivateLink — a regional Hyperplane event degrades all three simultaneously.
Nitro Hypervisor / Nitro Card
AWS's hardware-offload substrate on every EC2 host. For Gateway Endpoints: intercepts packets whose destination matches the prefix list CIDRs and routes them to the internal fabric, tagging the request with aws:SourceVpce. For Interface Endpoints: recognises ENI IPs as Hyperplane endpoints and encapsulates the packet into the substrate tunnel rather than routing it normally.
aws:SourceVpce
An IAM condition key set by the AWS fabric (Nitro for Gateway Endpoints, Hyperplane for Interface Endpoints) on requests arriving through a VPC endpoint. Used in S3 bucket policies, KMS key policies, SQS queue policies, etc., to restrict access to requests from a specific endpoint ID. Requires private DNS enabled on Interface Endpoints; absent if the endpoint's DNS override is disabled.
Andromeda (GCP)
Google Cloud's software-defined networking fabric, implementing Virtual Private Cloud networking and Private Service Connect. Fully distributed; no centralized network appliances in the data path.
Delta Sharing Network Gateway
A Databricks-managed HTTPS proxy (announced Summer 2025, in preview) that serves as an intermediary for Delta Sharing traffic when the provider's storage is fully private. Recipients route their data reads through the gateway, eliminating the need for direct storage access.
