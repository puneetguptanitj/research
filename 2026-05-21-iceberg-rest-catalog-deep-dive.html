The Iceberg REST Catalog (IRC)

An open HTTP/JSON API specification that decouples query engines from catalog implementations — and, in doing so, turns the catalog into the lakehouse's authorization decision point and credential broker.

Prepared 2026-05-21 Reading time ~24 min Focus Architecture · Landscape · Auth & credential vending · Connections Spec baseline Iceberg 1.11.0 (released 2026-05-19)

TL;DR

The Iceberg REST Catalog (IRC) is not a product — it is a protocol. It is the OpenAPI-defined HTTP contract in the apache/iceberg repository that any engine and any catalog can implement, so that Trino, Spark, Flink, DuckDB, Snowflake and PyIceberg all talk to Polaris, Unity Catalog, Gravitino, Lakekeeper or Nessie through one wire format. Before the IRC, the "catalog" was a thick client library (Hive Metastore Thrift, Glue SDK) wired into every engine; the IRC collapses that into a thin HTTP client and moves the intelligence — commit conflict resolution, capability negotiation, and security — server-side. It sits at L3, the catalog layer of the lakehouse stack, directly above the Iceberg table format and below the query engines, and it is the point where the technical catalog and the security control plane converge.

  1. The IRC is the contract, not the catalog. "Polaris" or "Lakekeeper" is an implementation; the IRC is the spec they conform to. Engines depend only on the spec, which is what makes catalogs swappable.
  2. Commits became a server-side transaction. The client sends a list of requirements (optimistic-concurrency assertions) plus updates; the server validates atomically and either commits or returns 409. The catalog — not the object store — is now the serialization point for ACID.
  3. Credential vending is the headline security feature. On loadTable, the catalog returns short-lived, path-scoped storage credentials (or pre-signed URLs via remote signing). Engines stop holding long-lived cloud keys; the catalog brokers every storage grant.
  4. The catalog is now the Policy Decision Point. Because every table load and commit flows through the IRC, the catalog can authenticate the caller (OAuth2/OIDC), evaluate RBAC/ABAC, and scope the vended credential — making L3 the natural home for governance that L8 used to bolt on at the edges.
  5. Convergence is real but governance is not standardized. The IRC standardizes the metadata API. It does not standardize the RBAC model, policy language, or credential-vending semantics — each implementation still differs, which is the single biggest portability gap as of 2026.
  6. The ecosystem consolidated in early 2026. Apache Polaris became a Top-Level Project (Feb 2026) and Project Nessie is being merged into it; the IRC is now the de facto universal catalog interface rather than one option among many.

1What it is & core architecture

The Iceberg REST Catalog is a vendor-neutral, HTTP/JSON catalog protocol defined by a formal OpenAPI document — open-api/rest-catalog-open-api.yaml in the apache/iceberg repository — and implemented client-side by the RESTCatalog / RESTSessionCatalog classes. It exists because Iceberg deliberately separates the table (the on-disk format spec) from the catalog (whatever tracks the pointer to a table's current metadata file). For years that "whatever" was a fat client: a Hive Metastore Thrift client, the AWS Glue SDK, a JDBC driver — each one a different dependency, a different security model, a different bug surface, baked into every engine. The IRC replaces all of them with a single thin REST client and a server that any vendor can build behind.

Architecturally the IRC has three moving parts: a thin client embedded in the engine, a catalog service that owns the metadata and the policy decisions, and a metadata/credential boundary where the service hands the engine just enough to read object storage directly. The deliberate design choice is that the catalog service sits on the control path (resolve, authorize, commit) but not the data path — bulk Parquet bytes never transit the catalog. That split is what lets a single small catalog service front petabytes.

What the spec actually defines

The OpenAPI document is concrete and worth knowing by its parts. It covers six endpoint families, all under /v1/{prefix}/, and crucially it makes most of them optional and discoverable:

Endpoint familyRepresentative routesNotes
Config / discoveryGET /v1/configReturns server defaults, overrides, and the list of supported endpoints — the basis of capability negotiation.
Namespace opsGET/POST /v1/{prefix}/namespaces, HEAD …/{ns}Create, list, load, drop; HEAD for cheap existence checks.
Table ops…/namespaces/{ns}/tables, POST …/tables/{t}Create, load, drop, rename, register; POST on a table is the commit.
View ops…/namespaces/{ns}/views/{v}Same lifecycle for Iceberg views; optional, advertised separately.
Transactions & stagingPOST …/transactions/commitMulti-table atomic commit; staged create supports atomic CTAS.
Server-side scan planningPOST …/tables/{t}/plan, …/tasksOptional: the catalog plans the scan and returns file tasks, off-loading manifest reads from the engine.

The discovery mechanism deserves emphasis. A client first calls GET /v1/config; the server replies with the exact set of endpoints it supports (e.g. "POST /v1/{prefix}/namespaces/{namespace}/tables/{table}/plan"). If the list is absent, the client falls back to a hard-coded default set (RESTSessionCatalog.DEFAULT_ENDPOINTS). This is how the spec evolves without breaking old clients: new capabilities ship as new advertised endpoints, and a client that does not understand them simply never calls them.

Key takeaway

The IRC's design centre is the relocation of intelligence: commit conflict resolution, capability negotiation, and security all move from a fat per-engine client into one server. The engine keeps only a thin HTTP client and direct storage I/O. Every downstream property — swappable catalogs, central governance, credential vending — falls out of that one decision.

2Placement in the data platform & AI landscape

The IRC sits at L3, the catalog layer: directly above the Iceberg table format (L2) and the object storage / file formats beneath it (L0–L1), and directly below the query and compute engines (L4). What makes its placement interesting is that it does not stay in its lane. Because every table resolution and every commit passes through it, the IRC has become the place where L3 fuses with the identity plane (L8) — it authenticates callers and makes authorization decisions — and increasingly with the governance plane (L7), since policy authored in a knowledge catalog has to be enforced somewhere, and the IRC is the only chokepoint that sees every access.

In replacement terms: the IRC replaces the Hive Metastore and the per-engine Glue/JDBC catalog clients as the engine-facing interface. It complements the Iceberg table format (the IRC is how you reach Iceberg tables, not a substitute for them) and the knowledge/governance catalog (which authors policy the IRC enforces). It does not compete with object storage, query engines, or the semantic layer — those are different layers. The one genuine competitive tension is with proprietary catalog-plus-engine bundles (a warehouse's built-in catalog): the IRC's whole pitch is that the catalog should be an open, swappable interface rather than a lock-in point.

Naming hazard

"Catalog" is overloaded. The IRC is a technical/operational catalog — it tracks tables, mediates commits, vends credentials. A knowledge/governance catalog (Atlan, Collibra, Purview, DataHub) does discovery, glossary, lineage and stewardship. They are converging — Unity Catalog is deliberately both — but a sentence about "the catalog" should always say which one it means.

3How it works — the lifecycle of a read and a commit

The IRC's mechanism is best understood through two operations: resolving and reading a table, and committing a change to it. The read path shows the control/data split; the commit path shows why moving conflict resolution server-side mattered.

First, config negotiation (steps 1–2) is not ceremony: the server's overrides can force settings the client cannot countermand, and the advertised endpoints list tells the client which optional features — views, server-side planning — are actually available. Second, server-side scan planning (the optional POST …/plan endpoint, matured through the Iceberg 1.10/1.11 line) lets the catalog read the manifests and return ready file-scan tasks. That moves manifest I/O off the engine — useful for thin clients like PyIceberg or a serverless function — at the cost of putting more work, and more trust, on the catalog.

3.2 Committing a change — the server-side transaction

This is the part the IRC genuinely re-architected. With the Hive Metastore, conflict resolution was a client-side dance: read current metadata, build a new metadata file, attempt an atomic compare-and-swap of the pointer, and on failure re-read and retry — all logic living in the engine. The IRC turns the commit into a declarative server-side transaction. The client sends a single POST on the table containing two lists:

  • Requirements — optimistic-concurrency assertions the server must validate before applying anything: e.g. assert-table-uuid, assert-ref-snapshot-id (the branch still points where the client thinks), assert-current-schema-id, assert-default-spec-id.
  • Updates — the declarative metadata changes to apply: add-snapshot, set-snapshot-ref, add-schema, set-current-schema, remove-snapshots, set-properties, and so on.

The server validates all requirements atomically. If any fails — because a concurrent writer moved the ref — it returns 409 Conflict and nothing is applied. The client can rebuild its updates against the new state and retry. The serialization point is now the catalog, not an object-store conditional write.

Watch out

A 409 is safe to retry; a 5xx on a commit is not. After a 5xx the client cannot know whether the pointer swap succeeded before the error, and blindly retrying risks committing the same change twice or corrupting state. The Iceberg client (1.10.0 onward) was fixed to not auto-retry commit POSTs after 5xx — your catalog must therefore make commits idempotent or surface a definitive outcome. Treat this as a design requirement, not an edge case.

4Authentication, authorization & credential vending

This section is the reason the IRC matters beyond plumbing. The IRC turns the catalog into three things at once: an authentication relying party, a Policy Decision Point (PDP), and a credential broker. Keep the three distinct — authN proves who is calling, authZ decides what they may do, and credential vending issues the storage grant to act on it.

4.1 Authentication — how a caller proves identity

The IRC's primary authentication mechanism is OAuth2 bearer tokens. The Iceberg client supports three concrete flows, all visible in OAuth2Util / OAuth2Properties:

  • Client-credentials grant. The client holds a credential of the form client_id:client_secret, exchanges it at a token endpoint for a short-lived access token, and sends that as Authorization: Bearer on every call. This is the service-to-service path — a Trino cluster authenticating to a catalog.
  • Bearer token (pre-issued). A token is supplied directly; no exchange happens. Common when an external IdP already minted the token.
  • Token exchange (RFC 8693). A subject token carrying user context is exchanged — authorized by an actor token (the catalog/service credential) — for a narrower access token. This is how a multi-tenant engine preserves the end-user's identity through to the catalog rather than collapsing everyone into one service principal.

In production, the catalog is almost always wired to a real OIDC provider (Okta, Entra ID, Keycloak) via its discovery document, so tokens are standard JWTs the catalog validates against the IdP's JWKS. Lakekeeper, for example, takes a LAKEKEEPER__OPENID_PROVIDER_URI and validates externally issued tokens directly. One important note for new builders: the spec's built-in v1/oauth/tokens endpoint — the catalog acting as its own toy authorization server — is deprecated and was always intended for development; real deployments delegate to an external IdP.

4.2 Authorization — the catalog as PDP

Once the caller is identified, the catalog evaluates an authorization policy on every namespace, table, and view operation. Here the IRC spec stops and implementations diverge — this is the standardization gap. The dominant patterns:

  • Two-level RBAC (Polaris). A principal holds principal roles; principal roles are granted catalog roles; catalog roles carry fine-grained privilege grants on securables (catalog, namespace, table, view, policy). The indirection is what keeps grants manageable across many principals.
  • ReBAC via OpenFGA (Lakekeeper). Lakekeeper externalizes authorization to OpenFGA — a Google-Zanzibar-style relationship engine — whose bidirectional inheritance maps naturally to nested namespaces. For engine-side checks it ships an OPA bridge that translates OpenFGA relations into Open Policy Agent input, so Trino can ask one consistent authority.
  • Identity-inheriting vending (Unity Catalog). UC's IRC and Hive endpoints vend credentials that inherit the privileges of the requesting principal, so the same governance applies whether the caller arrives via UC's native API or the IRC.

Beyond coarse RBAC, mature catalogs add ABAC-style controls — row filters, column masking, tag-based policy — evaluated at the catalog and, increasingly, expressed by returning a restricted view or by handing the engine policy it must apply. The reasonable read: RBAC handles the coarse "can you see this table" grant, ABAC handles "which rows/columns", and the IRC is the enforcement seam for both.

4.3 Credential vending — the mechanism in detail

Credential vending is what makes the catalog a security control point rather than a directory. The problem it solves: without it, every engine needs standing cloud credentials broad enough to read the lake — long-lived IAM keys sprayed across clusters, a credential-sprawl and blast-radius nightmare. With vending, engines hold no storage credentials. They authenticate to the catalog; the catalog, which holds the real cloud trust, mints a narrow grant per table load.

The client signals what it can accept with the X-Iceberg-Access-Delegation request header, whose two standardized values define two distinct trust models:

Delegation modeWhat the catalog returnsTrust & scope
vended-credentials Short-lived cloud credentials in the LoadTableResponse config — STS session tokens (AWS), SAS tokens (Azure), or GCS access tokens — scoped to the table's storage prefix. Engine briefly holds a real, time-bound, path-scoped credential. Fewer round-trips; the engine reads storage directly until the token expires.
remote-signing Nothing reusable. For each object request the engine calls the catalog's signing endpoint, which returns a pre-signed request for that one file and operation. The tightest model — the engine never holds a storage credential at all. The cost is a signing round-trip per file, so it favours security over throughput.

Watch out

Vended credentials are scoped and short-lived by design — and engines must actually honour both properties. Real 2025–2026 Trino issues show the failure modes concretely: clients that ignored the vended credential in the LoadTableResponse and fell back to instance credentials; clients that never re-called the credentials endpoint to refresh before expiry, breaking long-running queries; and uneven coverage across S3 vs. Azure SAS. Before standardizing on vended credentials, verify your specific engine version refreshes them and supports your cloud — the spec being correct does not mean every client implements it correctly.

Key takeaway

The IRC collapses three jobs onto the catalog: authenticate the caller (OAuth2/OIDC, token exchange for delegation), authorize the action (RBAC/ABAC as PDP), and vend a narrow storage grant (vended credentials or remote signing). That is the entire security argument for the IRC — and also its entire risk surface: the catalog is now a single point of trust whose availability and correctness gate the whole lake.

6Trade-offs, failure modes & when (not) to use it

The IRC is, on balance, the right default for a new Iceberg lakehouse in 2026 — but its strengths come with a sharply different operational profile from a file-system or metastore catalog, and an honest assessment has to name what it concentrates.

What it genuinely buys you

  • Real engine portability. The same Iceberg table is reachable from Trino, Spark, Flink, DuckDB, Snowflake and PyIceberg through one protocol. Swapping the catalog implementation does not touch the engines.
  • Correct, server-side commits. Conflict resolution stops being replicated (and subtly mis-implemented) per engine. The catalog is one serialization point with one definition of a valid commit.
  • Credential blast-radius collapse. Engines stop holding long-lived cloud keys. A leaked credential is now short-lived and path-scoped instead of a standing key to the lake.
  • One enforcement seam. AuthN, authZ, and audit converge on a single chokepoint — far easier to reason about than policy scattered across engine configs and IAM.

Where it breaks — failure modes to design against

  • The catalog is a single point of trust and availability. If the IRC is down, no engine can resolve or commit a table — the lake is effectively offline. Catalog HA, latency, and throughput are now first-class platform concerns, not afterthoughts.
  • Governance is not standardized. The IRC standardizes the metadata API, not the RBAC model, policy language, or vending semantics. Migrating from Polaris RBAC to Lakekeeper's OpenFGA model is a real re-implementation. "Open protocol" does not mean "portable policy."
  • Client maturity is uneven. As the 2025–2026 Trino issues show, a conformant spec does not guarantee a conformant client: vended credentials ignored, not refreshed before expiry, or unsupported on some clouds. Always validate the exact engine version against your storage backend.
  • Commit ambiguity after 5xx. A server error on a commit leaves the outcome genuinely unknown; the platform must make commits idempotent or surface a definitive result, and operators must understand that retries are unsafe here.
  • The catalog can become a hot path. High-concurrency workloads with many small operations — or aggressive server-side scan planning — push real load onto the catalog. It is a service to capacity-plan, not a passive directory.

When it fits

  • A multi-engine Iceberg lakehouse where the same tables are read and written by Trino, Spark, Flink, and Python clients.
  • Any environment where standing cloud credentials on compute is unacceptable and you need centralized, short-lived, scoped storage access.
  • Platforms that need one auditable enforcement point for table access — including AI agents that must be governed identically to human callers.

When to avoid it or wait

  • A single-engine, single-team setup with no portability or central-governance need — the operational cost of running an HA catalog service may exceed the benefit.
  • Workloads that depend on a specific catalog's advanced governance and would treat the IRC's portability as theoretical — be honest that you are buying the implementation, not the protocol.
  • When your chosen engine's IRC client cannot yet correctly handle vended credentials for your cloud — verify before committing the architecture, not after.

7Architect's take

The IRC has effectively won the catalog-interface argument. By early 2026 — with Apache Polaris a Top-Level Project and Nessie folding into it — the question for a new Iceberg platform is no longer "REST catalog or not" but "which implementation, and how do I run it well." Four things actually matter for the decision:

  • Choose the implementation for its governance model, not its protocol. Every option speaks the same IRC. The real differentiator is the authorization model — Polaris's two-level RBAC, Lakekeeper's OpenFGA/ReBAC, Unity Catalog's principal-inheriting vending. That choice is the one you cannot cheaply reverse, because the IRC does not make policy portable.
  • Treat the catalog as tier-1 infrastructure. It is now on the critical path of every query and every commit. Budget for HA, latency SLOs, capacity planning, and a tested failure story. A catalog outage is a lake outage.
  • Make credential vending real, not nominal. The security win only lands if engines hold no standing keys, honour scope, and refresh tokens before expiry. Validate this end-to-end on your exact engine versions and cloud — the gap between spec and client implementation is where this quietly fails.
  • Plan for the catalog as the AI governance point. As agents become first-class data consumers, the IRC is the natural place to govern them — same RBAC, same audit, same vended scope as a human. An MCP-style agent that discovers tables through the catalog inherits its policy for free.

What to watch over the next 12–18 months: (1) whether the community standardizes any part of the governance layer — a portable policy or vending contract would close the IRC's biggest gap; (2) the maturation of server-side scan planning, which shifts work and trust further onto the catalog and changes its capacity profile; (3) the Polaris/Nessie consolidation and whether catalog-level versioning (git-for-data) becomes a first-class IRC concept; and (4) the agent/MCP integration path — the catalog is positioned to become the control point for agentic access to the lake, and the implementations are already building toward it.

Glossary

IRC (Iceberg REST Catalog)
The OpenAPI-defined HTTP/JSON catalog protocol for Apache Iceberg; the contract engines and catalog implementations both conform to.
Credential vending
The catalog returning short-lived, path-scoped cloud storage credentials on a table load, so engines need no standing storage keys.
Remote signing
An alternative delegation mode in which the engine holds no credential and instead asks the catalog to pre-sign each individual storage request.
Requirements / updates
The two lists in an IRC commit: optimistic-concurrency assertions to validate, and the declarative metadata changes to apply.
PDP (Policy Decision Point)
The component that evaluates an authorization policy and returns permit/deny — a role the IRC catalog increasingly plays for table access.
Token exchange (RFC 8693)
An OAuth2 flow that swaps one token for a narrower one, used to carry an end user's identity through a service to the catalog.
ReBAC
Relationship-based access control (Google-Zanzibar lineage); the model OpenFGA, and thus Lakekeeper, uses for hierarchical namespace permissions.
