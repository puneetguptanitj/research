Twelve diagrams covering grant types, token flows, JWKS key distribution, key rotation, and OIDC vs OAuth2

OAuth2 is a framework with pluggable grant types. JWT Bearer is just one of them.

4. Mental Model — Separation of Concerns

OAuth2 is the protocol, JWT is the format. They operate at different layers.

The JWKS endpoint returns a JSON array of public keys. The kid field links each key to JWTs that reference it in their header.

7. Zero-Downtime Key Rotation via JWKS URI

When the auth server sees an unknown kid, it re-fetches the JWKS endpoint to pick up new keys — no re-registration needed.

9. The openid Scope — What Triggers OIDC

OAuth2 and OIDC use the same /token endpoint. The presence of scope=openid is the switch that flips OAuth2 into OIDC — and the only meaningful difference in what comes back is the ID Token.

flowchart TD A["Authorization / Token Request"] --> B{"scope includes\n'openid'?"} B -->|"No — plain OAuth2"| C["Returns:\naccess_token only"] B -->|"Yes — OIDC"| D["Returns:\naccess_token + id_token"] C --> E["Access Token\n───────────────\nsub = client / service ID\nscope = what it can do\nNo human identity claims"] D --> F["Access Token (same as left)\n+\nID Token ← always a JWT\n───────────────\nsub = stable human user ID\nemail, name, email_verified\nauth_time, nonce\nWho the human IS"] style B fill:#fef3c7,stroke:#d97706,color:#78350f style C fill:#dce8fb,stroke:#2f6fb8,color:#1a1d24 style D fill:#d1fae5,stroke:#10b981,color:#065f46 style E fill:#f0f5ff,stroke:#2f6fb8,color:#1a1d24 style F fill:#ecfdf5,stroke:#10b981,color:#065f46

10. ID Token vs Access Token — Two Different Questions

Scopes answer what can this token do? — that's authorization. ID Token claims answer who is the human behind this? — that's authentication. They coexist but are never the same thing.

flowchart LR subgraph AT ["Access Token (OAuth2)"] direction TB A1["Question answered:\n'What is permitted?'"] A2["Key claims:\nscope = read:data write:reports\nsub = service-account-42\niss = auth server\nexp = short TTL"] A1 --- A2 end subgraph IT ["ID Token (OIDC only — always a JWT)"] direction TB I1["Question answered:\n'Who is the human?'"] I2["Key claims:\nsub = user-abc123\nemail = alice@example.com\nname = Alice Smith\nemail_verified = true\nauth_time = when they logged in\nnonce = replay protection"] I1 --- I2 end AT -->|"both returned\ntogether in OIDC"| IT style AT fill:#dce8fb,stroke:#2f6fb8,color:#1a1d24 style IT fill:#fef3c7,stroke:#d97706,color:#78350f style A1 fill:#bfdbfe,stroke:#2f6fb8,color:#1e3a8a style I1 fill:#fde68a,stroke:#d97706,color:#78350f

11. RFC 7523 (JWT Bearer) in the Full Landscape — M2M vs Human

RFC 7523 is designed for machine-to-machine scenarios where no human is present. OIDC is designed for human authentication. The two axes — how the client authenticates and whether a human is present — are orthogonal.

flowchart TD ROOT["Requesting an Access Token"] --> HU{"Human\npresent?"} HU -->|"Yes — user in browser"| AC["Authorization Code Flow\n(browser redirect + login screen)"] HU -->|"No — machine / service"| M2M["Machine-to-Machine (M2M)"] AC -->|"+ scope=openid"| OIDC["OIDC\nReturns access_token + id_token\nApp knows WHO the user is"] AC -->|"no openid scope"| OA2["Plain OAuth2\nReturns access_token only\nDelegated access, no identity"] M2M --> CC["Client Credentials\ncredential = client_secret\n(shared password — weaker)"] M2M --> JB["JWT Bearer — RFC 7523 ✦\ncredential = signed JWT assertion\n(asymmetric key — stronger)\nService accounts, CI/CD, microservices"] JB --> WHY["Why stronger?\nPrivate key never leaves client\nAuth server stores public key only\nNo shared secret to leak"] style OIDC fill:#d1fae5,stroke:#10b981,color:#065f46 style JB fill:#2f6fb8,color:#fff,stroke:#1a4d8f style WHY fill:#dce8fb,stroke:#2f6fb8,color:#1a1d24 style CC fill:#f6f7f9,stroke:#9ca3af,color:#374151 style OA2 fill:#f6f7f9,stroke:#9ca3af,color:#374151
